# Dev Log — 2026-04-18

## Fix: ChoCh flip-bar ordering bug (engine.py)

### Root cause
`InstrumentEngine.on_bar()` processes bar state in this order:
1. `_handle_st_flip()` — detects ST flip, pushes pivot, resets all hypothesis states
2. `_update_choch()` — detects ChoCh using `feature_row['st_dir']` as `entry_dir`

`feature_row['st_dir']` is the **post-flip** direction. So on a bar where the entry TF
ST flips from -1 → +1 **and** `close > last_high` (structural break = bullish ChoCh):
- Step 1 runs first: flip detected, states reset, `_last_st_dir` updated to +1
- Step 2 runs: checks `ctx_dir == +1 AND entry_dir == -1` — fails because `entry_dir` is now +1
- ChoCh missed entirely on that bar

Since ChoCh structural breaks and ST flips nearly always coincide (price breaking the
prior swing high is exactly what drives the ST flip), **91–97% of ChoCh opportunities
were being missed** across all phases.

Diagnostic counts (B-only, cap=1, before fix):
- Phase 1: 64 detected / 775 total → 91.7% missed, 36 B trades
- Phase 7: 13 detected / 421 total → 96.9% missed, 22 B trades
- Phase 8: 14 detected / 411 total → 96.6% missed, 21 B trades

### The fix
Capture `prior_entry_dir = self._last_st_dir` in `on_bar()` BEFORE calling
`_handle_st_flip()`. Pass it to `_update_choch()` as `prior_entry_dir`.
`_update_choch()` now uses `check_dir = prior_entry_dir if prior_entry_dir is not None
else entry_dir` for the structural break check.

This allows ChoCh detection on the flip bar (the prior direction was bearish even
though the current direction is now bullish). The **same-bar B entry block** in
`b_trigger()` is KEPT — B cannot enter on the ChoCh confirmation bar because entry
at bar open cannot know the bar will close as a ChoCh. Only ChoCh **detection** was
the bug; the same-bar entry block was correct.

Diagnostic counts (B-only, cap=1, after fix):
- Phase 1: 709 detected / 709 total → 0% missed, 701 B trades
- Phase 7: 367 detected / 367 total → 0% missed, 583 B trades
- Phase 8: 372 detected / 372 total → 0% missed, 570 B trades

### Implication for study
All Round 1 and Round 2 B results are invalid and must be re-run.
A2 volume gap (V7: 466 trades vs Policy 3 phase3: 42 trades) is under investigation
for a similar ordering issue.

### Files changed
- `backtester/engine.py` — `on_bar()` captures `prior_entry_dir`; `_update_choch()`
  accepts `prior_entry_dir` kwarg and uses it for the structural break direction check
- `EURUSD/diag_choch_count.py` — diagnostic script (keep for A2 investigation)

## Fix: A2 OF direction mode config flag (engine.py, strategy.py, EURUSD.yaml)

### Root cause
`_update_new_extreme` and `evaluate_hypotheses` both derived A2's OF direction from
`of_direction_from_pivot()` which requires a self-proved HH+HL or LH+LL pivot pattern.
This returns `None` when the pivot array is ambiguous or has insufficient history,
blocking A2 new_extreme arming and the OF gate entirely.

V7 used `dir_ctx` (context TF ST direction) — always +1 or -1, never None — giving
~466 A2 trades vs Policy 3's ~42 on the equivalent phase (~11x gap).

### The fix
New config flag `hypotheses.A2.of_direction_mode` (EURUSD.yaml):
  - `lax`   — OF direction = context TF ST (1H/4H/etc, phase-agnostic). Always defined.
               The 4/2 OF gate still runs — it checks the pivot array *in that direction*.
               Matches V7 behavior.
  - `strict` — OF direction must be self-proved by pivot HH+HL or LH+LL pattern.
               Ambiguous or insufficient pivots → skip trade. Original behavior.

Current setting: `lax`

### Implementation
- `engine._update_new_extreme(feature_row, context_row)` — added `context_row` param.
  Reads `of_direction_mode` from config; lax path uses `int(context_row['st_dir'])`.
- `engine.on_bar()` — passes `context_row` to `_update_new_extreme`.
- `strategy.evaluate_hypotheses()` — `of_dir` resolved via same flag before computing
  `a2_direction` and `a2_gate`. Lax path uses already-computed `ctx_dir`.
- `EURUSD.yaml` — `hypotheses.A2.of_direction_mode: lax` with inline comment.

### Files changed
- `backtester/engine.py`
- `backtester/strategy.py`
- `backtester/EURUSD.yaml`

## Feature: A1 st_touch trigger mode (strategy.py, EURUSD.yaml)

### Finding
V7 A1 trigger fires on **ST line touch** (± 3 pip epsilon), not EMA20.
Policy 3 only had `ema20_touch` and `ema20_rsi_v2` — no ST line variant existed.

Comparison:
| Mode       | Touch target | Clean approach | EMA3 lookback |
|------------|-------------|---------------|---------------|
| ema20_touch | EMA20       | ✓ required    | lag2 > current (2-bar) |
| st_touch    | ST line ±3pip | ✗ none      | lag2 > current (2-bar) |

Volume expectation: `ema20_touch` > `st_touch` because EMA20 is closer to price
and touched on shallower pullbacks, despite the extra clean_approach filter.
ST line sits deeper (12–20 pip below in trend) and is reached less often.

### Implementation

- Added `st_touch` case to `a1_trigger()` dispatcher
- YAML `trigger_mode` comment updated to list all three modes
- Current config: `trigger_mode: ema20_touch` (unchanged)

## Audit: flicker_suppression independence (strategy.py)

### Finding
`_flicker_allowed(engine, hypothesis)` is fully parameterized — reads per-hypothesis
config, calls `engine.has_open_hypothesis(hypothesis)` dynamically. No hardcoded names.

Each hypothesis calls it with its own name:
- A1: `_flicker_allowed(engine, 'A1')`
- A2: `_flicker_allowed(engine, 'A2')`
- B:  `_flicker_allowed(engine, 'B')`

All three are independent — an open B or A2 trade does NOT block A1 (own_hypothesis mode).
This is intentional Policy 3 concurrent behavior, distinct from V7 which used global
suppression (`len(open_trades) == 0`).

### Fixes applied
1. Removed pre-computed `a1_touch_allowed` variable — inlined `_flicker_allowed(engine, 'A1')`
   directly in the `if` condition. Old name was misleading (implied touch-only but applied
   to all trigger modes including `st_touch`).
2. A1 flicker check now matches A2 and B style (all inline).

### Files changed
- `backtester/strategy.py`

## Audit: Cross-phase runner (split_cursor.py + policy3_cross_phase.py)

### Design verified correct
- **Signal clock is branch-native**: `_events()` builds only from `branch.aligned.index`
  (entry-TF bar closes — 5m or 15m). The 1m feed never generates signal events. ✅
- **Fill before signal ordering**: each event: `_fill_pending_due` → exits → `on_bar`.
  A pending trade fills before the current bar's `evaluate_hypotheses` runs, so
  `_flicker_allowed` and `has_open_hypothesis` correctly see the open trade. ✅
- **Signal state independence**: ChoCh, pivot_array, new_extreme_flag, sb_used, cooldowns
  all live on the individual branch engine. A1 branch does not pollute B branch. ✅
- **Shared open_trades**: all branch engines point at `SplitCursor._shared_open_trades`.
  `has_open_hypothesis`, stacking checks, `_check_shared_exits` all read the same list. ✅
- **Entry price is branch-native**: `_queue_for_next_open` looks up the next row from the
  branch's own aligned dataframe — uses the correct TF bar open. ✅
- **Same-bar priority B > A2 > A1**: `policy_3` sorts candidates by priority before
  iterating. Insertion order of branches does not affect outcome. ✅
- **No signal spam**: `accept_candidate` for B sets `sb_used`, blocking re-entry.
  For A1/A2, the trade fills before the next on_bar call (fill-before-signal), so
  `_flicker_allowed` blocks re-entry correctly. ✅
- **Pullback gate per-context**: each candidate carries its own branch's `context_dir`
  (A1 uses 1H, B uses 4H). `is_pullback_trade` is evaluated per-hypothesis correctly. ✅

### Bug 1 — FIXED: Cross-phase cap violation (pending entries not counted in cap check)

`policy_3` checked `len(engine.open_trades)` for the position cap. In cross-phase,
a pending entry (accepted but not yet filled) is not in `open_trades` — it lives in
`SplitCursor._pending` until the branch's next bar close.

**Attack scenario (cap=1, A1 on 5m, B on 15m):**
```
Event T   : B fires, A1 fires → B wins priority → B queued (fills at T+15m)
Event T+5m: A1 fires → cap check sees open=0, pending_B invisible → A1 accepted!
Event T+10m: A1 fills → 1 open trade
Event T+15m: B fills  → 2 open trades  ← CAP=1 VIOLATED
```

**Fix (3-file change):**
- `SplitCursor._pending_slots: int` counter tracks queued-but-unfilled entries
- `_queue_for_next_open` increments it on successful queue
- `_fill_pending_due` decrements it (before opening trades)
- `_inject_effective_count()` patches `execution._effective_open_count = open + pending`
  into the config dict passed to `policy_fn`
- `policy_3` reads `_effective_open_count` (falls back to `len(engine.open_trades)`
  for single-phase cursor where this key is never set)

Single-phase `cursor.py` is unaffected — `_effective_open_count` is never in its config,
so `policy_3` falls back to the existing `len(engine.open_trades)` path.

### Bug 2 — NOTED: `profile_run(bar_duration_min=1)` gives wrong bar-based metrics

Cross-phase runs have no single entry TF. Bar-based metrics (holding_bars,
inter_arrival_bars, utilization, natural_N) are expressed in 1m bar units.
Calendar-minute metrics (holding_min, inter_arrival_min) are still correct.

Added comment in `policy3_cross_phase.py` documenting this. When comparing cross-phase
natural_N against standalone profiles, convert: `bars_1m / 5 → 5m-bars`, `bars_1m / 15 → 15m-bars`.

### Bug 3 — FIXED: Dead condition in `_queue_for_next_open`

`get_indexer([event_time])` always returns a length-1 array. The guard
`len(positions) != 1` was always False. Removed. Only `positions[0] < 0` is meaningful.

## Feature: A2 and B SL params wired to YAML config (strategy.py, EURUSD.yaml)

### Background
A2 uses pivot-clamped SL: `sl = max(sl_min, min(sl_max, pivot_extreme + sl_epsilon))`.
B uses fixed SL. Both had these values hardcoded in module-level constants with no YAML
override path. EURUSD uses tighter caps than other symbols (A2 cap 20pip vs 25pip;
B fixed 20pip vs 25pip).

### SL mode taxonomy
| Hypothesis | Mode | EURUSD value | Other symbols |
|------------|------|-------------|---------------|
| A1         | `fixed_12pip` | 12pip flat | `ema20_st_pocket` (variable) |
| A2         | `pivot_clamped` | floor 15pip, cap 20pip, buf 3pip | cap 25pip |
| B          | `fixed` | 20pip flat | 25pip flat |

### Changes

**`a2_sl_tp()`** — new signature:
```python
def a2_sl_tp(direction, entry_price, pivot_array,
             sl_min=HYP_A2_SL_MIN, sl_max=HYP_A2_SL_MAX,
             sl_epsilon=HYP_A2_SL_EPSILON, rr=HYP_A2_RR) -> tuple[float, float]:
```
`rr` param used in TP calculation (was hardcoded).

**`b_sl_tp()`** — new signature:
```python

            sl_fixed=HYP_B_SL_FIXED, rr=HYP_B_RR) -> tuple[float, float]:
```

**`evaluate_hypotheses()`** — candidate dicts now stash SL params at signal time:

## Refactor: session_gate — label-based + raw-hour-range system (strategy.py, EURUSD.yaml)

### Problem
`is_in_session()` used two hardcoded module constants (`LONDON_START_UTC = 7`,
`NY_END_UTC = 21`) — a single fixed window with no per-hypothesis or per-symbol
granularity. JPY pairs need Asian session included; custom kill-zone windows were
impossible without code changes.

### Design
Two YAML formats, both resolved by `_session_allowed()`:

**Format 1 — raw hour ranges** (`allowed_hours`, inline):
```yaml
hypotheses:
  A1:
    session_gate: true
    allowed_hours: [[7, 11], [12, 13], [21, 5]]
```
Each sub-list is `[start, end)`. Wrap-around supported: `[21, 5]` → 21:00–05:00 UTC
(overnight crossing midnight). Use for one-off or per-hypothesis windows.

**Format 2 — named labels** (`sessions:` + `allowed_sessions`):
```yaml
sessions:
  asian:     [0,  7]
  london:    [7,  12]
  london_ny: [12, 17]
  ny:        [17, 21]
  my_window: [21, 5]    # custom names allowed

hypotheses:
  A1:
    session_gate: true
    allowed_sessions: [london, my_window]
```
Use for windows shared across multiple hypotheses or symbols.

Priority: `session_gate: false` → `allowed_hours` → `allowed_sessions`.
`allowed_hours` takes precedence if both are present.

### Implementation

- Removed `LONDON_START_UTC`, `NY_END_UTC`, `is_in_session()`
- Added `DEFAULT_SESSIONS` dict (canonical 4-label fallback)
- Added `_sessions_cfg(config)` — reads `config['sessions']` or falls back to `DEFAULT_SESSIONS`
- Added `session_label_for(bar_time, sessions)` — maps UTC hour to label; catch-all → `'asian'`
- Replaced `_session_allowed()` with dual-format implementation:
  - Format 1: `any(_in_range(h, s, e) for s, e in allowed_hours)`
  - Format 2: `session_label_for(bar_time, sessions) in allowed_sessions`
- Old stale duplicate of `_session_allowed()` removed

### EURUSD.yaml changes
- Added top-level `sessions:` block (4 canonical labels)
- Added `allowed_sessions: [london, london_ny, ny]` to A1, A2, B
  (inactive while `session_gate: false` — ready for Step 4 sweep)

### Files changed
- `backtester/strategy.py` — removed old constants + `is_in_session()`; added `DEFAULT_SESSIONS`,
  `_sessions_cfg`, `session_label_for`, new `_session_allowed`
- `backtester/EURUSD.yaml` — top-level `sessions:` block; `allowed_sessions` per hypothesis
- `docs/session-killzone.md` — updated with label table, both YAML formats, wrap-around example
