Phase E — HTF Expansion Monitoring

TL;DR — Phase E is the classifier's waiting state while the HTF P/D leg is actively expanding. The classifier tracks a running extreme, watches for the two-bar reaction signal that opens Phase D, and runs an internal shadow node machine (E.seeking → E.stalling → E.pullback_developing) to detect slow pullbacks without waiting for a full Phase D confirmation. It emits a watching / skip hypothesis — no entry, just monitoring.
→ Back to 401 Hypothesis DAG → Related: 311 Evidence Compiler

00 Core Concept

Phase E is the HTF trend expansion in progress. The market has broken structure, the P/D leg is open, and price is making new extremes in the bias direction. The classifier enters Phase E when no counter-evidence is ready yet — it is explicitly waiting for the trend to exhaust itself enough to offer a trade.

Phase E emits a watching hypothesis with entry_policy = "skip". It is not tradeable. Its job is to track the running extreme and feed the shadow node machine so that D and C transitions are correctly timed.

What Phase E is

  • A tracking state — consumes EC's running high (long) or running low (short) watermark for the current HTF P/D leg.
  • A signal consumer — reads EC two-bar reaction and reaction-failed facts used by _phase_d_setup() and the D/C collapse paths.
  • A shadow state machine — internally advances E.seeking → E.stalling → E.pullback_developing to detect slow counter-moves before they fully appear on the HTF.

What it is not

  • Not a trade entry phase. entry_policy = "skip" always.
  • Not a gating condition — the classifier enters Phase E simply because no D/C/B condition is met while the HTF phase is open or pullback_confirmed with a valid bias.
  • Not an interpreter of LTF orderflow internals. EC owns the phase_e_context facts; Layer 4 only consumes the compiled flags needed by the DAG and shadow node machine.
Phase E entry: HTF bias is directional, HTF phase is open, and no D/C/B gate fires. EC refreshes the running extreme on every Phase E bar. The classifier reads those compiled facts and hands the shadow node result to the emitted hypothesis's phase_sub_status.

01 EC Watermark

The running Phase E high/low watermark now belongs to the Evidence Compiler. EC emits the current watermark and reaction facts through the phase_e_context candidate. Layer 4 no longer keeps a classifier-local active_phase_e_extreme_price or active_phase_e_extreme_time.

Layer 4 still keeps active_phase_e_direction as lightweight origin memory for DAG guards such as _previous_or_active_e(). That field records only whether the classifier has an active Phase E direction; it does not own the price watermark.

Discretionary → Algorithm mapping

EC watermark facts phase_e_context

EC updates its Phase E watermark before emitting phase_e_context. The classifier reads the resulting debug facts instead of recomputing the same watermark from flat HTF structure fields. 

phase_e_context.debug_facts:
  active_phase_e_direction
  active_phase_e_extreme_price
  active_phase_e_extreme_time
  new_htf_extreme
  reaction_failed
  reaction_failed_rule
New extreme check — did this bar push past the watermark? EC → DAG

EC emits new_htf_extreme. Layer 4 consumes that flag directly. It no longer calls a local _new_phase_e_extreme() helper.

This flag is consumed in the shadow node machine (pd_expanding resets to E.seeking), in Phase D setup through EC's htf_counter_reaction candidate, and in B-branch resolution back to E.

$$ \text{new\_extreme}_\text{long} = \text{EC.phase\_e\_context.new\_htf\_extreme} $$
EC owns price/time watermark state. Layer 4 owns the DAG decision and only keeps active_phase_e_direction as origin memory. Do not reintroduce classifier-local Phase E price/time tracking.

02 Two-Bar Reaction Signal

Every bar in Phase E (and in Phase D/C branches that check for reaction failure), _phase_e_reaction() compares the current HTF bar against the previous one to detect whether the expansion is being threatened.

Two outputs — reaction_warning and reaction_confirmed — gate the D-phase entry. The distinction matters: a warning bar (wick below prior low, close above) is ambiguous; a confirmed bar (close through prior low) is the trigger.

Discretionary → Algorithm mapping

Warning vs confirmed — wick through vs close through _phase_e_reaction()

For a bullish HTF trend:

prev_low  = previous_htf_bar["low"]
cur_low   = current_htf_bar["low"]
cur_close = current_htf_bar["close"]

reaction_warning   = cur_low < prev_low              # wick below — ambiguous
reaction_confirmed = cur_close < prev_low            # close below — D-phase trigger

For a bearish HTF trend (mirrored):

reaction_warning   = cur_high > prev_high
reaction_confirmed = cur_close > prev_high
Reaction-failed check — D-phase tested but price kept going phase_e_context

EC emits reaction_failed when price closes past the previous Phase E watermark, meaning the counter-move was absorbed and the trend resumed. When the previous phase is D or C, Layer 4 reads this EC fact directly. 

direction == "long":
  reaction_failed = phase_e_context.reaction_failed

direction == "short":
  reaction_failed = phase_e_context.reaction_failed

When reaction_failed is true the classifier re-enters Phase E without refreshing a local watermark. EC already owns the updated watermark.

$$ \text{reaction\_confirmed}_\text{long} = \text{close} < \text{prev\_low} \quad\wedge\quad \text{reaction\_failed}_\text{long} = \text{close} > \text{E\_extreme} $$
reaction_warning is a heads-up only — the classifier does not change phase on warning alone. reaction_confirmed gates Phase D entry. reaction_failed gates the forced return to Phase E from D or C. These three flags are always computed together and put in debug_facts.

03 Shadow Node Machine

The shadow node machine runs inside every Phase E bar via _phase_e_shadow_facts(). It is a lightweight internal classifier — tracking whether the E-phase expansion is still seeking new highs, has stalled, or is developing a slow counter-move — without needing to promote to Phase D first.

The current node is stored in phase_e_shadow_node on HypothesisClassifierState. It flows into the emitted hypothesis as phase_sub_status (e.g., "seeking", "stalling", "pullback_developing") and is the key used by one Phase C transition path ("slow pullback from E").

E.seeking
→stopped expanding
E.stalling
→LTF counter MSS confirmed
E.pullback_developing
any node
→new HTF extreme (pd_expanding)
E.seeking

E.seeking

The default node. The HTF P/D leg is actively making new extremes this bar (pd_expanding = True). Any node resets to E.seeking when a new extreme arrives — the pullback attempt was absorbed. Also the initial node when Phase E is entered fresh (previous phase was not E, or direction changed).

E.stalling

No new HTF extreme this bar. The expansion has paused. The machine waits here until either a new extreme appears (→ E.seeking) or EC confirms an LTF counter MSS from a live probe break of the protected orderflow anchor (→ E.pullback_developing).

E.stalling flow — pause before proven pullback EC → DAG → shadow

Discretionary read: Phase E was pushing the HTF P/D level, then price stopped extending that level. The chart may start probing around the P/D boundary, but the pullback has not proven itself yet. 

EC facts:
  phase_e_context.status == "ready"
  new_htf_extreme == false
  htf_pd_stopped_expanding == true
  ltf_bias_counter_htf = true/false        # useful evidence, not required
  ltf_probe_outside_htf_pd_range = true/false  # useful evidence, not required
  reaction_confirmed == false              # otherwise E can exit toward D

DAG gate:
  previous E shadow node == E.seeking
  no new HTF extreme
  no higher-priority E→D exit fired
  => keep phase == E
  => ask Phase E shadow to move into E.stalling

Phase E shadow memory:
  node = E.stalling
  previous_node = E.seeking
  bars_in_node starts / increments
  no orderflow interpretation is performed by the shadow

The important boundary: LTF bearish/counter movement is precise EC evidence, but it should not be required for the first E.seeking → E.stalling transition. Requiring it would skip the quiet pause and make E.stalling mean "pullback already visible", which belongs to the next node.

A failed LTF pro attempt is not enough by itself. Without EC's confirmed LTF counter MSS flag, the machine stays in E.stalling; the failed attempt is only debug context.

E.pullback_developing

EC confirmed an LTF counter MSS while the HTF was stalling: a live counter probe broke the protected HL/LH while orderflow was still officially pro-HTF. This uses phase_e_context_ltf_counter_orderflow_mss_watch, not the standalone structural ltf_counter_choch stream, because structural ChoCh is too fast and noisy for this shadow boundary. The node holds until a new HTF extreme resets to E.seeking or the D gate fires from the EC. There is no E.pullback_developing → E.stalling return path; a failed pro-continuation attempt from this node belongs to Phase D, not back to stalling.

E.pullback_developing flow — LTF counter orderflow MSS before Phase C EC → DAG → shadow

Discretionary read: price reached the Phase E peak and stopped extending. EC then sees a live counter probe break the protected LTF orderflow anchor while orderflow is still officially pro-HTF. That compiled MSS flag is the only shadow trigger; a full counter-directional orderflow verdict belongs to later C.slow maturity. A single structural ChoCh remains useful chronology for Phase D liquidity, but is no longer enough for this Phase E node. 

EC facts (phase_e_context candidate):
  htf_pd_stopped_expanding == true
  new_htf_extreme == false
  ltf_counter_orderflow_mss_watch == true  ← entry trigger
    orderflow.confirmed_direction == pro-HTF
    orderflow.mss_regime == "mss_watch"
    orderflow.mss_monitor_status == "watching_resolution"
    orderflow.mss_trigger_source == "probe_vs_protected_anchor"
    orderflow.probe_breaks_protected_anchor == true
  ltf_pullback_depth_pct = number | null   # supporting metric
  reaction_confirmed == false              # otherwise D can win

DAG gate:
  previous E shadow node == E.stalling
  ltf_counter_orderflow_mss_watch == true
  no higher-priority E→D exit fired
  => keep phase == E
  => ask Phase E shadow to move into E.pullback_developing

Phase E shadow memory:
  node = E.pullback_developing
  previous_node = E.stalling
  bars_in_node starts / increments
  no orderflow internals are re-interpreted by the shadow

Exits from E.pullback_developing:
  new HTF extreme (pd_expanding)  →  E.seeking  (expansion resumed)
  D gate ready                    →  Phase D    (failed pro attempt)
  neither                         →  hold E.pullback_developing

Phase C is the next maturity step and should be reviewed now that the same orderflow flip is again the E.pullback_developing entry trigger. The current runtime still promotes to slow C when ltf_counter_bos_confirmed is true from ltf_counter_story.

When phase_e_shadow_node == "E.pullback_developing" and ltf_counter_bos_confirmed is true (LTF orderflow confirmed_direction == counter-HTF and regime == "directional"), the classifier promotes directly to Phase C with sub_status = "slow_pullback", bypassing Phase D entirely. This is the "slow pullback from E" path.

The shadow node is reset to E.seeking on every epoch boundary (via _reset_phase_e_shadow()). It is not reset on phase transitions — if the market goes D→C→B and then resolves back to E, the shadow restarts from E.seeking only because the epoch ID has changed, or because previous_phase != "E" at re-entry (which triggers selection_reason = "phase_e_shadow_initialized").

04 Exits from Phase E

Phase E can exit to Phase D (primary), Phase C via fast or slow pullback, or to Phase A finale (when previous_phase == "A" — the leg has already reached the HTF objective). All three checks run on every bar while the HTF is open or pullback_confirmed.

Discretionary → Algorithm mapping

→ Phase D: normal reaction path classify() main branch

Gate conditions (all must hold):

  1. Not from B or A: previous_phase not in {"B", "A"} — prevents re-entry from a resolved trade setup from immediately triggering D.
  2. Previous or active E: previous_phase == "E" or active_phase_e_direction == direction — ensures D is only opened when there is an E-phase extreme to react from.
  3. Phase D ready: phase_d["phase_d_ready"] — the three-part D gate from _phase_d_setup(): HTF stopped expanding + HTF opposing SD + LTF counter SD (or liquidity variant).

When all three hold, the classifier emits Phase D with a sub_status derived from the trigger type (reaction_point, htf_pd_grab_reclaim_test, htf_eq_grab_reclaim_test).

→ Phase C: fast hard pullback through midpoint classify() fast E→C branch

A compressed shortcut for tight P/D ranges and choppy markets. If the HTF leg opens Phase E but the LTF probe immediately pulls through the P/D midpoint with clean counter orderflow, the classifier can skip the slow E subnode walk and emit C.hard_pullback.

  1. D first: the normal Phase D gate is checked before this shortcut.
  2. No new HTF extreme: phase_e_context_new_htf_extreme == false.
  3. Clean counter orderflow: ltf_counter_orderflow_clean == true and ltf_counter_orderflow_broken == false.
  4. Midpoint breach: ltf_pullback_depth_pct >= 50.

The emitted Phase C hypothesis carries phase_c_sub_status = "hard_pullback", phase_c_origin_node = "E.fast_hard_pullback", phase_c_selection_reason = "fast_midpoint_pullback_after_phase_e_expansion", and phase_c_quality = "choppy_tight_range_fast_pullback".

→ Phase C: slow pullback path from E.pullback_developing classify() E.pullback_developing branch

An alternative exit that bypasses Phase D entirely. Conditions:

  1. Not from B: previous_phase != "B".
  2. Shadow node: phase_e_shadow_node == "E.pullback_developing" — EC confirmed LTF counter MSS while HTF was stalling.
  3. LTF orderflow BoS: ltf_counter_bos_confirmed — LTF orderflow confirmed_direction == counter-HTF and regime == "directional" (Case F level). Counter bias alone is not sufficient.

The emitted Phase C hypothesis carries debug keys: phase_c_origin_node = "E.pullback_developing", phase_c_sub_status = "slow_pullback", phase_c_selection_reason = "ltf_pd_flipped_counter_after_failed_e_continuation_attempt". Status is armed if a counter POI exists, otherwise watching.

→ Phase E (self): no exit condition met classify() fallback

When HTF phase is open and no D/C/B condition fires, the classifier re-emits Phase E and records the active E direction. This is the normal bar-by-bar hold path during active expansion.

Warm-state pullback_confirmed also holds Phase E when the previous Layer 4 phase is already E and no explicit D/C/B/C exit fires. The emitted debug includes phase_e_hold_reason = "pullback_confirmed_without_explicit_exit". Cold-start self-location from none(warmup) remains owned by the none phase note.

Phase D and Phase C are checked in strict order: D is checked first. If D fires, C is not evaluated. The fast hard-pullback C path is a midpoint-depth shortcut for tight ranges; the slow-pullback C path only activates when phase_e_shadow_node == "E.pullback_developing" and the D gate has not fired. This means a fully-developed D condition always wins over C.

05 Algorithm

# classify() — Phase E entry and exit checks (simplified)
if phase in {"open", "pullback_confirmed"} and direction != "none":
    reaction = self._phase_e_reaction(direction, current_bar, previous_bar)
    debug.update(reaction)

    # ── Previous phase was A (A-finale check) ──
    if self.state.previous_phase == "A":
        finale = self._phase_a_finale(direction, htf, current_bar)
        if finale["phase_a_finale_closed_beyond"]:
            hyp = self._phase_e(direction, cursor_time, debug, ltf)
            return self._commit(hyp)

    # ── D and C exit gates ──
    phase_d = self._phase_d_setup(snapshot, htf, ltf, direction, current_bar)
    if (
        self.state.previous_phase not in {"B", "A"}
        and self._previous_or_active_e(direction)
        and phase_d["phase_d_ready"]
    ):
        hyp = self._phase_d(direction, cursor_time, {...})
        return self._commit(hyp)

	    phase_c_from_e = self._phase_c_setup(snapshot, ltf, direction)
	    fast_hard_pullback = self._phase_c_fast_hard_pullback_setup(debug, phase_c_from_e)
	    if (
	        self.state.previous_phase != "B"
	        and fast_hard_pullback["phase_c_fast_hard_pullback_ready"]
	    ):
	        hyp = self._phase_c(direction, ..., phase_sub_status="hard_pullback")
	        return self._commit(hyp)

	    if (
	        self.state.previous_phase != "B"
	        and self.state.phase_e_shadow_node == "E.pullback_developing"
        and phase_c_from_e["phase_c_ltf_counter_bos_confirmed"]  # Case F: confirmed_direction + regime directional
    ):
        hyp = self._phase_c(direction, ..., phase_sub_status="slow_pullback")
        return self._commit(hyp)

    # ── Fallback: stay in Phase E ──
    if phase == "open" or (
        phase == "pullback_confirmed" and self.state.previous_phase == "E"
    ):
        if phase == "pullback_confirmed":
            debug["phase_e_hold_reason"] = "pullback_confirmed_without_explicit_exit"
        hyp = self._phase_e(direction, cursor_time, debug, ltf)
        return self._commit(hyp)
# Shadow node state machine — runs inside every Phase E bar
def _phase_e_shadow_facts(self, ltf, direction, debug):
    previous_node = self.state.phase_e_shadow_node
    pd_expanding  = bool(debug.get("phase_e_context_new_htf_extreme"))

    # Phase E waits for EC's LTF counter MSS flag.
    # Standalone structural ltf_counter_choch remains available for D/liquidity chronology.
    ltf_counter_orderflow_mss_watch = bool(
        debug.get("phase_e_context_ltf_counter_orderflow_mss_watch")
    )

    # Transition table
    if self.state.previous_phase != "E" or self.state.active_phase_e_direction != direction:
        selected_node = "E.seeking"          # fresh entry
    elif pd_expanding:
        selected_node = "E.seeking"          # new extreme → reset
    elif previous_node == "E.seeking":
        selected_node = "E.stalling"         # no new extreme
    elif previous_node == "E.stalling":
        if ltf_counter_orderflow_mss_watch:
            selected_node = "E.pullback_developing"  # counter orderflow MSS watch opened
        else:
            selected_node = "E.stalling"
    elif previous_node == "E.pullback_developing":
        selected_node = "E.pullback_developing"   # hold; D gate fires independently

    self.state.phase_e_shadow_node = selected_node
    phase_sub_status = selected_node.split(".", 1)[1] if "." in selected_node else None
    return {"phase_sub_status": phase_sub_status, "phase_e_shadow_node": selected_node, ...}

07 Open Issues · Edge Cases · Next Milestone

Edge Case

No previous HTF bar on first tick. _phase_e_reaction() returns all-false when previous_bar is None. On the very first bar the replayer processes, the two-bar signal is silent. The shadow node also initialises to E.seeking regardless. This is correct but means Phase D cannot fire on bar 1.

Edge Case

Shadow node resets on direction flip within same epoch. If HTF bias flips from bullish to bearish within the same epoch ID (rare but possible in range conditions), active_phase_e_direction != direction triggers selected_node = "E.seeking". The shadow transition still depends only on the node, new-extreme status, and EC's compiled LTF counter MSS flag.

Edge Case

E.pullback_developing has no timeout decay. The node holds indefinitely once EC's LTF counter MSS flag moves the shadow into E.pullback_developing. There is no bar-count timeout, and the transition table has no return path to E.stalling. Only a new HTF extreme resets to E.seeking, or the D gate fires. A bar-count threshold could be added if the node proves to over-persist.

Out of Scope

reaction_confirmation config key not read. config.yaml sets reaction_confirmation: close_through_previous_htf_bar but HypothesisClassifier.__init__() does not read it. The two-bar rule is hardcoded. If a "wick-through" variant is ever needed, this key would be the hook.
