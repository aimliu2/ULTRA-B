# 2026-04-21 Backtester Speed Pass

## Goal

Optimize the current `backtester` version without rewriting the strategy,
engine, or policy layers into another language.

The preferred direction was:

- keep the Python codebase
- use Numba for numeric loops where it fits
- avoid a large cursor/engine rewrite for now

## Bottleneck Review

Profiling a 5k-bar Policy 3 phase run showed the main bottleneck was not
strategy logic itself. The slow parts were mostly execution-walker and pandas
overhead.

Baseline profile shape:

- `cursor._exit_via_1m()` was the largest cost.
- The cursor was slicing the 1m dataframe and iterating with pandas `iterrows()`
  for every decision bar.
- This happened even when there were no open trades.
- Per-bar pandas `Series` slicing/renaming for `feature_row` and `context_row`
  was also expensive.
- Indicator loops, especially `st_step_count`, were another measurable cost.

## Changes Made

### 1. Added Numba Kernels

Added `backtester/numba_kernels.py` for numeric indicator kernels:

- EMA
- ATR
- RSI
- SuperTrend
- ST stable flag
- ST step count

These are array-in / array-out kernels, which is the right shape for Numba.
The strategy/engine logic remains Python.

Note: `cache=True` was intentionally not used because this OneDrive-backed
workspace caused Numba cache locator errors. Numba still works; it just compiles
inside the active Python process.

### 2. Replaced Indicator Loops With Numba-Backed Versions

Updated:

- `backtester/indicator.py`
- `backtester/features.py`

The biggest conceptual cleanup was `st_step_count`.

Old behavior:

- for each bar, walk backward through the current ST episode
- this can become expensive on long episodes

New behavior:

- compute step count forward in one linear pass
- same intended result, less repeated work

### 3. Optimized 1m Exit Scan

Updated:

- `backtester/cursor.py`
- `backtester/split_cursor.py`

The new rule:

```text
if no open trades: skip 1m scan entirely
if open trades: use numpy timestamp arrays + searchsorted
```

This is safe because the current backtest protocol uses the 1m clock for
exit precision after a trade exists. Entry still follows the existing rule:

```text
signal at closed entry-TF bar
fill at next entry-TF bar open
```

So skipping the 1m scan when there are no open trades cannot miss an exit.

Important distinction:

This does not implement "fill pending entries at next 1m open". If that becomes
the desired protocol later, pending entries must also be checked by the 1m clock.
That would be an execution-model change, not just an optimization.

### 4. Reduced Per-Bar Pandas Row Churn

Updated:

- `backtester/cursor.py`
- `backtester/split_cursor.py`

Previous cursor path:

```python
feature_row = _extract_feature_row(row)
context_row = _extract_context_row(row)
```

That created new pandas `Series` objects and renamed context fields on every
bar.

Current path:

```python
feature_row = row
context_row = _ContextRow(row)
```

`feature_row` is still the pandas `Series` returned by `iterrows()`.
`_ContextRow` is a lightweight view that maps:

```text
context_row["st_dir"] -> row["ctx_st_dir"]
```

This avoids a major rewrite while removing avoidable allocation.

## Current Optimization Level

The codebase is now at **Level 1.5**, not Level 2 or Level 3.

### Level 1.5: Current

- still uses pandas `iterrows()`
- still passes a pandas `Series` as `feature_row`
- uses lightweight `_ContextRow` for context access
- uses Numba for indicator kernels
- uses numpy arrays/searchsorted for 1m exit scanning

### Level 2: Not Implemented

Would preload dataframe columns as numpy arrays and use an `ArrayRow` wrapper:

```python
arrays = {
    "close": df["close"].to_numpy(),
    "st_dir": df["st_dir"].to_numpy(),
}

for i in range(n):
    feature_row = ArrayRow(arrays, i, timestamp)
```

This would avoid pandas `Series.__getitem__` calls in the cursor loop, but it
would touch more of the row-provider path. Keep this as a future option if the
current speed is still not enough.

### Level 3: Not Implemented

Would rewrite engine/strategy/policy to operate directly on typed arrays and
numeric codes. This is faster but much more invasive and is not desired now.

Also smoke-tested the cross-phase runner after the cursor changes.

## OneDrive Note

OneDrive sync likely made long studies worse, especially because these runs
read parquet files and write many CSV/JSON outputs. Pausing sync helps, but the
profile showed real code-level bottlenecks too.

For heavy tournament runs, preferred setup:

- pause OneDrive sync, or
- run from a non-synced local workspace, or
- write result outputs to a non-synced folder

## Remaining Speed Frontier

The next meaningful speed frontier is replacing `iterrows()` with an
array-backed row provider. That would be a medium-sized cursor change, not a
full strategy rewrite.

Decision for now:

Keep the codebase at Level 1.5 and avoid bigger rewrites unless runtime still
hurts after the current optimization and OneDrive pause.
