00 Migration Boundary

This tracker keeps the open decisions visible while Layer 4 is rebuilt phase by phase. The target boundary is simple: EC emits facts, the DAG selects the phase node, and each phase shadow stores only the memory needed inside that phase.

Open migration rule: do not let a shadow thesis decide the global phase. Shadow state should explain the current phase's internal status, while the DAG remains responsible for phase transitions.
Next Milestone

Define phase-owned shadow objects. Decide whether each phase gets a small dataclass such as PhaseEShadow, DShadow, and BShadowThesis, instead of one generic ShadowThesis that quietly means B.

Edge Case

Stop legacy priority from leaking back in. The old classifier searched E, D, C, B, A by priority. The migration needs an explicit rule for when that priority is still valid, and when the DAG must hold the active node.

Fallback

Prefer one active phase during rebuild. If cross-phase behavior becomes noisy, run the classifier with only E enabled first, then add D, C, B, and A back in order.

01 none / warmup

none is still the silence state, not a trading phase. The open work is making sure it only means warmup, no closed HTF structure, or genuinely no trade, instead of catching valid post-warmup phase states.

Open Issue

Allow warmup to self-locate into any current phase. Once the first usable HTF structure exists, the DAG should be able to move from none(warmup) into E, D, C, B, or A from current EC/DAG evidence instead of only supporting none → E.seeking.

Edge Case

Separate silence reasons. Keep warmup silence, missing 15m/4H close, no structure, and no trade as distinct none_sub_status values so a valid E origin does not disappear under the catch-all.

Fallback

Debug blocked transitions. If a bar stays none while EC has enough E facts, emit a blocked-transition reason such as blocked_none_to_e.

02 Phase E · Closed

Phase E migration is closed. Price remains in expansion monitoring while the HTF P/D level is extending, stalling, or developing a clean counter pullback. Only an explicit destination-phase contract may move the DAG out of E.

The completed ownership split is explicit. EC owns observable facts, Layer 4 owns phase choice, and the Phase E shadow thesis owns only the internal E.seeking / E.stalling / E.pullback_developing memory.

Migration status: Closed. There are no remaining Phase E migration items in this ledger. References to E inside none, D, and C describe the origin boundary for those destination phases; they do not reopen Phase E migration. Lower-risk Phase E runtime oddities remain documented as edge cases in 402 Phase E. 
EC owns:
  - phase_e_context candidate, emitted when HTF direction is known
  - active_phase_e_extreme_price / active_phase_e_extreme_time
  - new_htf_extreme
  - HTF stopped-extending fact
  - LTF probe outside current HTF PD range
  - HTF close outside current HTF PD range
  - standalone LTF counter structural ChoCh for D/liquidity chronology
  - clean unbroken LTF counter orderflow as Phase E pullback-developing trigger and fast-C context
  - LTF pullback depth / displacement as supporting metrics, not the first trigger
  - reaction_warning / reaction_confirmed / reaction_failed
  - raw evidence refs and debug facts explaining those observations

Layer 4 DAG owns:
  - whether current hypothesis phase is none, E, D, C, B, or A
  - none/warmup -> current best phase transition once usable HTF structure exists
    (E, D, C, B, or A)
  - holding E when LTF probes outside PD range before HTF close
  - E -> D transition only on the agreed counter-reaction contract
  - E.stalling -> E.pullback_developing when LTF counter orderflow MSS / P-D flip appears
  - E -> C slow-pullback transition only after shadow reports pullback_developing and the LTF counter story matures
  - disabled-phase blocked-transition debug during phase-by-phase replay

Phase E shadow thesis owns:
  - current E shadow node: E.seeking, E.stalling, or E.pullback_developing
  - previous E shadow node and bars_in_node
  - source LTF counter orderflow leg id / started-at, when retained
  - consumed source id so one orderflow MSS event does not double-fire
  - phase_sub_status output derived from the current E shadow node

Phase E shadow thesis must not own:
  - global phase selection
  - HTF/LTF fact detection
  - D/C/B/A transition priority
  - raw snapshot reads

03 Phase D

D should be reintroduced only after the E boundary is clean. The open work is defining the exact evidence that turns expansion monitoring into counter-reaction monitoring.

The proposed failed-continuation path is a sibling of the existing HTF reaction-point and liquidity-test paths. Its provisional node is D.continuation_failure_test: a first clean counter leg was disrupted by a distinct pro-aligned continuation leg, then a fresh clean counter leg appeared before any strict new HTF extreme. The second counter leg is the D origin event; the intervening unresolved interval remains E.stalling.

Current implementation note: Phase D now uses the simplified D.watch / D.speculation model documented in 402-hypothesis-phD.html. The legacy Phase D selector and old reaction/liquidity sub-status tests are intentionally quarantined for now. I will delete the legacy Phase D code/tests later after replay validation.
Next Milestone

Define the failed-continuation E-to-D contract. Add a phase-neutral EC candidate such as ltf_pro_continuation_failure that records the first counter leg, distinct pro-aligned continuation leg, fresh failure counter leg, and whether a strict new HTF extreme occurred before failure. Layer 4 should select D.continuation_failure_test only when those EC facts match Phase E shadow lineage.

Migration Issue

Do not treat every broken counter leg as a pro-continuation attempt. Current ltf_counter_orderflow_broken also covers compression, sweep-range, and broken-quality regimes. EC needs an objective pro-aligned leg fact before Layer 4 can arm the continuation-failure path.

Migration Issue

Track explicit leg lineage in phase-owned shadow memory. Replace an ambiguous pullback-cycle count with first-counter, pro-continuation, and failure-counter leg IDs plus consumed state. The shadow records Layer 4 history; EC remains unaware of E.pullback_developing and other DAG nodes.

Current Behavior Conflict

Revisit new_htf_extreme_with_ltf_counter_bias as a D trigger. A strict new HTF extreme normally proves successful Phase E continuation and should return or hold E.seeking. It should open D only when a separate explicit liquidity or reaction footprint justifies that transition.

Current Behavior Conflict

Keep reaction_failed separate from continuation failure. The current fact means an existing D/C reaction collapsed because price resumed through the Phase E extreme. It must not be reused to mean that a resumed pro-continuation attempt failed before creating a new extreme.

Edge Case

Handle probe versus close. A wick/probe against the expansion level and a closed HTF counter-reaction should not both trigger the same D origin.

Fallback

Keep D disabled during E-only replay. Until E stops falling to none, D should be unavailable or logged as blocked so its legacy branch cannot hide the E bug.

Deferred · Layer 4

D.speculation → E.stalling on SL hit is unimplemented. This is a Layer 4 phase transition — not a Layer 5 trigger decision. Layer 5 has no memory and must not own phase invalidation routing. Two implementation options when D is coded: (a) new EC candidate that detects price reclaim above choch_2.level while HTF is not above PD range; or (b) direct Shadow Thesis comparison in the DAG — current_price vs the committed choch_2["level"] stored in phase_d_shadow (allowed: price read for committed-level boundary check, not raw structural reasoning). Parked until D.watch / D.speculation are coded and passing in replay.

Fixed · 2026-06-09

EC candidate direction not validated at DAG read-time — fixed. d_choch_candidate.get("direction") == direction now gates both the D.watch entry gate and the D.watch hold block before consuming choch_seen / sb_seen.

Fixed · 2026-06-09

ltf_counter_sb_event_at added to EC. _compile_ltf_counter_choch() now emits ltf_counter_sb_event_at alongside ltf_counter_sb_seen and ltf_counter_sb_level. Path B freshness check uses it.

Verified Root Cause · Replay 2026-06-09

D.watch fired immediately after E.pullback_developing because Orderflow MSS and Structure ChoCh shared the same structural lineage. In EURUSD 15m/4h at 2026-01-28T15:30Z, orderflow MSS opened when the live probe broke the protected low-side HL at 1.19583. At 2026-01-28T16:00Z, the LTF structure engine closed below that same 1.19583 floor and emitted itrSbDown / structure_choch. The event was fresh, so timestamp freshness gates passed, but semantically it was the close-confirmation of the same MSS disruption that opened E — not a post-E D setup.

Agreed fix: split Orderflow into a separate orderflow_anchor_sequence / orderflow_probe store, add EC provenance fields for both MSS and structural ChoCh, and make D.watch require a distinct post-E pro-HTF attempt before consuming a later fresh counter structural event.

04 Phase C

C is the slow pullback and reaction-development phase. Its migration should make the path from E.pullback_developing or D explicit, without requiring B's shadow state to be alive.

Next Milestone

Re-review the slow C origin. Runtime gate is currently ltf_counter_bos_confirmed — LTF orderflow confirmed_direction == counter-HTF and regime == directional. Because E.pullback_developing now enters on MSS watch, slow C can remain the later confirmed-direction maturity step.

Next Milestone

Block direct E-to-B walking. E.pullback_developing should not move directly to B.initiation_watch. The pullback must first mature into C; only then can B evaluate a pro-side reclaim or initiation attempt.

Edge Case

Separate D-to-C from E-to-C. A confirmed counter-reaction after D and a slow pullback directly from E may share evidence, but the DAG should log which origin created C.

Fallback

Do not depend on B shadow. During the C pass, keep B disabled so C cannot appear correct only because B's current ShadowThesis absorbed ambiguous bars.

05 Phase B

B is currently the only extracted shadow thesis, but that object is really B-specific. The clean migration path is to gate it off, rename or reshape it, and bring it back after E, D, and C are explicit.

Next Milestone

Disable B while rebuilding earlier phases. Add a config or classifier gate so B cannot take priority while E, D, and C contracts are being verified in the replayer.

Next Milestone

Rename generic ShadowThesis. If the fields remain B-only, rename it to BShadowThesis or split it into phase-owned state before adding D/C/A shadow logic.

Edge Case

Split initiation from shallow reclaim. The B migration still needs a crisp boundary between the B setup candidate and the shallow-reclaim memory that survives across bars.

Fallback

Reset rules need proof in replay. B shadow should define exactly when it survives, resets, or gets consumed by A, with debug fields showing the reason.

06 Phase A

A should be last because it is the trade-objective phase. The open question is not just how to detect A, but how to prove it came from the right upstream phase.

Next Milestone

Migrate _phase_a_setup() off raw snapshots. A setup needs an EC candidate or an explicit deferral note before it can be trusted inside the DAG.

Edge Case

Define A origin from B. The DAG should reject an A candidate that does not have a valid B lineage, unless a separate exceptional origin is deliberately defined.

Fallback

Revalidate A finale after B returns. Even if _phase_a_finale() already has EC wiring, it should be replay-checked after B shadow rules are renamed and gated back in.
