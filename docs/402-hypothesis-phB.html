Hypothesis Phase B — B.watch

TL;DR — Phase B is a single node: B.watch. It fires when a pro-HTF macro ChoCh (last_sc) fires while in C.pullback AND the pullback depth has reached the discount zone (≥ 51%). At entry, commitment_extreme_level is locked to ltf_struct.range_low (long) / range_high (short) — the commitment extreme of the pullback. Exits (priority order): new HTF extreme → E.seeking; pro-HTF BoS → A.watch; commitment extreme breached → C.pullback; counter MSS → C.pullback. Direct E → B is DAG-blocked; B entry requires C.pullback depth gate.
→ Phase C — Counter Pullback → Phase D — Watch → DAG diagram

00 Overview

Phase B is the first counter-pullback confirmation node. It opens when price has pulled back deep enough (discount zone, ≥ 51% of the P/D range) that a pro-HTF structural ChoCh fires — the pro side attempting to reclaim. B.watch is the D-symmetric counterpart: D.watch mirrors the pro side (E pullback confirmed → D.watch → bounce initiation), while B.watch mirrors the counter side (C pullback confirmed → B.watch → reclaim initiation).

DAG position:

E.seeking → E.stalling → E.pullback_developing
  → D.watch
    → C.pullback           ← counter MSS confirmed; pullback leg active
        depth < 51%:   → C.pullback_weaken   ← pro-HTF SC in premium zone
        depth ≥ 51%:   → B.watch           ← pro-HTF SC; discount zone reached
          → C.pullback      ← counter MSS fails the reclaim attempt (D-symmetric)
          → E.seeking       ← new HTF extreme invalidates the thesis
E must mature through C before B is permitted. Direct E → B is DAG-blocked by _phase_b_blocked_by_dag(). The depth gate (≥ 51%) is enforced at the C → B transition boundary, not by the EC candidate.

01 Entry Gate — Depth Gate from C.pullback

B.watch opens from the C block (previous_phase == "C") when a pro-HTF macro ChoCh fires on a fresh leg AND the pullback has reached the discount zone. No additional EC candidate is required — the gate reads directly from phase_e_context facts already present in debug.

Gate conditions

last_sc.breakDirection == pro-HTF direction     # pro macro ChoCh (SC01/SC02)
last_sc.eventTimestamp > c_shadow.entered_at     # freshness: SC after C.pullback opened
ltf_pullback_depth_pct >= 51.0                   # discount zone reached

When depth < 51%, the same pro-HTF SC fires C.pullback_weaken instead (still in premium zone, not deep enough for B). See Phase C docs for that path.

The depth gate uses the same phase_e_context facts already available in the C block — no additional EC candidate roundtrip. The gate is enforced by the DAG at the C → B boundary, not pre-filtered by the EC candidate.

Re-entry signal — A.watch_weaken_invalidated origin at extreme

When C.pullback was entered from A.watch_weaken (prior B and A both failed), the DAG still enters B.watch at depth ≥ 51%. It does NOT redirect to E.seeking. Instead, when depth ≥ 90% at that origin, b_shadow.at_extreme_entry is set to True. Layer 5 reads this flag and raises the entry bar (e.g. requires htf_sd_zone_tapped or liquidity_pool_run before permitting entry).

# Only when c_shadow.origin_node == "A.watch_weaken_invalidated"
depth >= 90%  →  B.watch  b_shadow.at_extreme_entry = True   # Layer 5 signal
depth < 90%   →  B.watch  b_shadow.at_extreme_entry = False  # normal
The DAG never suppresses B.watch based on depth alone. at_extreme_entry is a Layer 5 advisory signal — the DAG always enters B.watch when the depth gate fires, regardless of origin. First-pass B entries (origin != "A.watch_weaken_invalidated") always have at_extreme_entry = False.

02 Shadow — PhaseBShadow

B.watch carries commitment pointers only. Layer 3 EC snapshot is the authoritative source for zone levels, orderflow direction, and prices. Layer 5 resolves IDs from the EC snapshot at decision time. Shadow fields are never zone levels or prices — only IDs, booleans, and timestamps. One exception applies: see commitment_extreme_level below.

PhaseBShadow fields

entered_at                    str | None   # cursor_time of C → B transition (freshness floor)
htf_sd_zone_tapped            bool         # True once HTF pro-SD was tapped/resolved
htf_sd_zone_id                str | None   # pointer — Layer 5 resolves level via EC
htf_sd_zone_tapped_at         str | None   # cursor_time of first tap
ltf_sd_zone_tapped            bool         # True once a LTF pro-SD zone appears
ltf_sd_zone_id                str | None   # pointer — first zone_id from ltf_pro_sd_zone_ids
liquidity_pool_run            bool         # True once HTF ITR liquidity pool ran
liquidity_pool_id             str | None   # pointer — htf_itr_grab_reclaim_pool_id
commitment_extreme_level      float | None # ltf_struct.range_low (long) / range_high (short)
                                           # captured at C → B entry bar — invariant exception
commitment_extreme_event_id   str | None   # last_sc.event_id (the pro SC that opened B) — Layer 5 provenance
at_extreme_entry              bool         # True when origin=="A.watch_weaken_invalidated" AND depth>=90%
                                           # Layer 5 advisory — DAG does not block B on this flag

entered_at is set on the C → B transition bar and serves as the freshness floor for the B.watch → C.pullback MSS exit guard. The zone and liquidity fields are updated each bar by _update_phase_b_watch_shadow(), which reads from htf_b_phase_setup EC facts.

commitment_extreme_level is set once at the C → B entry bar from ltf_struct.range_low (long) or ltf_struct.range_high (short) at that exact moment. This captures the direction-relative pullback commitment extreme at the moment B.watch opens. It is never updated afterward. This is a permitted exception to the pointer-only invariant: ltf_struct.range_low/range_high ratchets on every new structural SC and will not retain B's original commitment extreme across B's lifetime. The committed value is a fixed historical reference used as the B.watch invalidation level — if price breaches it, B.watch exits to C.pullback (origin_node = "B.watch_commitment_extreme_breached"). Phase A also reads this value via b_shadow.commitment_extreme_level for A.watch_weaken invalidation.

def _update_phase_b_watch_shadow(self, snapshot, debug):
    s = self.state.shadow_thesis.phase_b
    _, b_facts = _ec_candidate(snapshot, "htf_b_phase_setup")

    # HTF SD zone tap (sticky once set)
    if (b_facts.get("htf_pro_sd_tapped") or b_facts.get("htf_pro_sd_resolved")) \
            and not s.htf_sd_zone_tapped:
        s.htf_sd_zone_tapped = True
        s.htf_sd_zone_id     = b_facts.get("htf_last_resolved_zone_id")
        s.htf_sd_zone_tapped_at = debug.get("cursor_time")

    # LTF SD zone (sticky, first only)
    ltf_zones = b_facts.get("ltf_pro_sd_zone_ids") or []
    if ltf_zones and not s.ltf_sd_zone_tapped:
        s.ltf_sd_zone_tapped = True
        s.ltf_sd_zone_id     = ltf_zones[0]

    # Liquidity pool run (sticky)
    pool_id = debug.get("htf_itr_grab_reclaim_pool_id")
    if pool_id and not s.liquidity_pool_run:
        s.liquidity_pool_run = True
        s.liquidity_pool_id  = pool_id
Shadow pointer fields accumulate monotonically — they are never cleared within a B.watch episode. All three categories (HTF SD zone, LTF SD zone, liquidity pool) are sticky: once set, they remain until the next epoch reset.

03 B.watch State Machine

B.watch is a single-node state. Each bar the classifier checks three exit conditions in priority order.

Exit 1 — E.seeking (new HTF extreme)

When a new HTF extreme appears, the reclaim thesis is invalidated and the cycle resets to E.seeking. This mirrors D.watch → E.seeking exactly.

phase_e_context_new_htf_extreme == True
  OR e_ctx.get("new_htf_extreme") == True
→ E.seeking   # HTF continuation resumes; B.watch invalidated

Exit 2 — A.watch (pro-HTF BoS, freshness-guarded)

When a pro-HTF structural SC fires after B.watch opened, price has escaped the B zone in the pro direction — the pro attempt is progressing. commitment_extreme_level was already locked at B.watch entry; it is carried forward unchanged.

last_sc.breakDirection == pro-HTF direction   # pro BoS (SC01/SC02)
AND last_sc.eventTimestamp > b_shadow.entered_at   # freshness guard
→ A.watch
  a_shadow.entered_at = cursor_time
  # b_shadow.commitment_extreme_level already set at B.watch entry — not changed here

Exit 3 — C.pullback (commitment extreme breached)

When the LTF structural range extreme breaches the commitment extreme recorded at B.watch entry, price has invalidated the reclaim attempt — the pullback is extending past B's invalidation level. Priority is higher than the MSS check.

ltf_struct.range_low < b_shadow.commitment_extreme_level   # long
ltf_struct.range_high > b_shadow.commitment_extreme_level  # short
→ C.pullback
  c_shadow.origin_node = "B.watch_commitment_extreme_breached"
  c_shadow.entered_at  = cursor_time
  debug["phase_b_commitment_extreme_breached"] = True
  debug["phase_b_commitment_extreme_level"]  = b_shadow.commitment_extreme_level

Exit 4 — C.pullback (counter MSS, freshness-guarded)

When a counter orderflow MSS fires on a leg that started after B.watch opened, the pro reclaim attempt failed — price broke structure against the bounce. This mirrors D.watch → C.pullback exactly.

ltf_counter_orderflow_mss_watch == True
AND ltf_counter_orderflow_started_at > b_shadow.entered_at   # freshness guard
→ C.pullback
  c_shadow.origin_node = "B.watch_mss_failed"
  c_shadow.entered_at  = ltf_counter_orderflow_started_at
Freshness guard explained hypothesis.py:711–731

The freshness guard prevents a counter MSS that was already active when B.watch opened from immediately firing the exit. The guard requires started_at (the timestamp the counter orderflow leg began) to be strictly after b_shadow.entered_at (the cursor_time of the C → B transition bar). An MSS that started on the same bar as B.watch entry — or before — is stale and blocked.

# Probe observation (2026-02-03, EURUSD 15m/4h):
#   B.watch entered at 03:45; commitment_extreme=1.17756 locked at entry
#   mss_started=03:45 on bars 1–18 → blocked by freshness guard
#   Fresh mss_started=10:30 appears; B.watch → C.pullback (B.watch_mss_failed) fires at 11:45

Hold (carry)

If neither exit fires, B.watch carries forward. Shadow is updated by _update_phase_b_watch_shadow() each bar, accumulating zone tap and liquidity observations. Layer 5 reads these from snapshot debug_facts for entry permission decisions.

B.watch has no internal sub-statuses. Layer 5 reads phase_b_shadow.* facts from the snapshot to determine entry quality (zone tapped, liquidity run, etc.). The DAG only tracks the two exit conditions.

04 DAG Block — Direct E → B

When the classifier is in a Phase E context and previous_phase != "C", B is hard-blocked. B.watch entry is only valid via the C.pullback depth gate. The guard prevents E.pullback_developing from skipping C entirely.

# hypothesis.py:926–931
def _phase_b_blocked_by_dag(self, direction):
    return self._previous_or_active_e(direction) and self.state.previous_phase != "C"

When blocked, the debug section emits informational facts but no transition fires:

phase_b_candidate         = True
phase_b_ready             = False
phase_b_blocked_by_dag    = True
phase_b_dag_blocked_reason = "direct_e_to_b_requires_c_origin"
C is not optional. D.watch → C.pullback or V1 (E.pullback_developing no-pro path) are the only valid C entries that then permit B.watch via the depth gate.

06 Old Model — Archived

The pre-rework Phase B had three sub-statuses driven by dedicated EC candidates. All are superseded by B.watch. The old code paths remain in hypothesis.py as dead methods for reference.

B.strict_reclaim / B.shallow_reclaim (archived) hypothesis.py:1297–1549 dead

Entry: _phase_b_setup() read from htf_b_phase_setup EC candidate. Required htf_pullback_context_ready, location buckets (strict_pd_half / shallow_pd_half), ltf_turns_back_toward_htf, htf_pro_sd_reaction. The 51% depth gate was not enforced at the C block — it was implicit via htf_pullback_context_ready.

B.shallow_reclaim state machine: tracked shadow status (active → contested → weakened) based on counter evidence and same-level returns. Shadow used flat ShadowThesis fields (opened_at, selected_poi_id, first_counter_mitigation_at, same_level_return_count). These flat fields remain in ShadowThesis for backward compatibility with dead method references.

Exit paths: B.shallow → D.watch (when htf_opposing_sd_reaction), B.shallow → E.seeking (new extreme). B.strict had no internal sub-status machine.

B.initiation_watch (archived) hypothesis.py:1089–1286 dead

Entry: _phase_b_initiation_setup() read from phase_b_initiation EC candidate (nested under b_initiation). Required valid C origin (legacy C sub-status names: C.htf_reaction_pullback, C.slow_pullback, etc.) and b_init.status == "ready".

Sub-statuses: active → decisive → stalled/contested → weakened. Driven by ITR grab/reclaim signals (opposite_itr_grab_seen, source_anchor_run_seen).

Exit paths: → C.pullback.no_followthrough (source anchor run), → C.pullback.after_inducement (weakened + HTF opposing resolved + LTF counter story). Both of these C sub-statuses are also removed from the DAG — they are now Layer 5 context read from snapshot facts.

C.pullback.after_inducement / C.pullback.no_followthrough (archived)

These were B.initiation_watch fallback exit C sub-statuses. With B.watch replacing initiation_watch, they no longer exist in the DAG. The qualitative information they encoded (was there an inducement sweep? did the B attempt fail?) is now Layer 5 context: Layer 5 reads phase_b_shadow.* facts (zone tapped, liquidity run, etc.) from the snapshot to make that distinction.

Out of Scope

Layer 5 B.watch entry permission. Layer 5 has not been built yet. When built, it will read phase_b_shadow.* IDs and resolve zone/orderflow details from the EC snapshot. The shadow pointer-only design is intentional — Layer 5 decides entry quality, not the DAG.

Edge Case

B.watch fires repeatedly within a C.pullback episode. After B.watch → C.pullback (failed), the DAG is back in C.pullback. If a subsequent pro-SC fires at depth ≥ 51%, B.watch opens again. The shadow is reset only on epoch boundary (new HTF P/D range), so accumulated zone tap observations carry across re-entries.

Edge Case

htf_b_phase_setup key names unverified in B.watch context. _update_phase_b_watch_shadow() reads htf_last_resolved_zone_id and ltf_pro_sd_zone_ids from htf_b_phase_setup facts. These key names were taken from the old model — if the EC emits different keys, shadow fields will be None (non-crashing, but empty). Low priority: shadow is debug-only for now.
