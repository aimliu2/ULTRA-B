Layer 3 · Node 11 · Evidence Compiler

Evidence Compiler — Qualified Candidate Emission

TL;DR — The Evidence Compiler is the final Layer 3 stage after Fusion. It reads the DualContextSnapshot bar-by-bar, accumulates multi-bar evidence chains, and emits typed EvidenceCandidate objects for Layer 4 (DAG Navigator) to consume. It asks "what did the market do?" — emitting facts with timestamps and source object refs, not phase decisions. Six patterns cover the full trade lifecycle from B-initiation through Phase-E expansion. Stateful only for the B-initiation 3-step chain and the Phase-E extreme tracker; all other patterns are per-bar gates.

00 Core Concept

The Evidence Compiler sits between Fusion (which prints the current-bar truth snapshot) and Layer 4 (which makes thesis decisions). Fusion is stateless — it answers "what is true now." The EC is stateful — it answers "what has the market been doing across bars, and what does that chain of evidence look like?"

The EC does not decide phase. It does not read hypothesis state. It reads consecutive fused snapshots and emits a list of EvidenceCandidate objects on every bar — one per active pattern. Layer 4 decides which candidates to act on.

Six patterns

htf_b_initiation      3-step chain: source ITR grab → opposite ITR grab → HTF decision zone
                      Stateful — accumulates across bars. Resets on epoch boundary.

htf_counter_reaction  Counter-reaction facts: HTF P/D stopped expanding + opposing SD tapped/resolved
                      + LTF counter SD created. Liquidity variant requires current grab/reclaim
                      + reclaim inside active P/D + post-reclaim LTF ChoCh + HTF close resolution.

ltf_counter_story     C-phase gate: LTF bias flipped counter to HTF direction.
                      Emits selected POI if a counter LTF SD zone is available.

htf_b_phase_setup     B-phase direct entry gate: LTF turned back toward HTF direction
                      + pro-trend LTF SD zone inside HTF P/D half.
                      Sub-types: strict (deep pullback) vs shallow (50–70% P/D).

htf_pd_objective      A-phase finale: current HTF bar touched the P/D range extreme.
                      Sub-flag: phase_a_finale_closed_beyond for confirmed close past objective.

phase_e_context       Always emitted when HTF direction is known. Tracks the running
                      E-cycle extreme, two-bar reaction signal, and reaction_failed flag.

Epoch boundary

The EC derives an epoch ID from the HTF structure's last_sc event — specifically the event timestamp, event code, break direction, and phase_start_ts. When this ID changes, a new HTF P/D leg has started. The B-initiation accumulator resets and the Phase-E extreme tracker clears. This ensures cross-bar chains never leak across legs. 

epoch_id = f"{last_sc.eventTimestamp}|{last_sc.eventCode}|{last_sc.breakDirection}|{phase_start_ts}"
EC reads from fused["context_snapshot"] (DualContextSnapshot) and HTF OHLCV bars. It does not re-read raw channel snapshots. It asks "what did the market do?" — it emits facts, not phase decisions. DAG Navigator holds previous_phase / current_node state and decides which phase applies.

01 EvidenceCandidate Schema

Every pattern emits a single EvidenceCandidate dataclass per bar. The schema is uniform across all patterns — the pattern field and debug_facts carry pattern-specific fields. 

# evidence_compiler.py
@dataclass
class EvidenceCandidate:
    candidate_id: str           # stable per epoch (B-init) or uuid per bar
    pattern: str                # one of the 6 pattern names
    status: CandidateStatus    # "collecting" | "ready" | "invalidated" | "expired"
    direction: Direction       # "long" | "short" | "none"
    timeframe: str | None       # reference_tf (HTF) or execution_tf (LTF)
    evidence_refs: list[EvidenceRef]           # typed pointers to market events
    source_object_refs: list[dict[str, Any]]   # zone_ids, pool_ids, level prices
    location_context: dict[str, Any]           # price, timestamp, pd_position
    blocked_reasons: list[str]                 # why status != ready
    first_seen_at: str | None   # when pattern first activated this epoch
    ready_at: str | None        # timestamp when status flipped to ready
    debug_facts: dict[str, Any] # pattern-specific raw flags for inspection
# A typed pointer to a specific piece of market evidence
@dataclass
class EvidenceRef:
    ref_id: str | None          # pool_id, zone_id, or None
    kind: str                   # "liquidity_pool" | "sd_zone" | "pd_level" | "structure_event"
    source: str | None          # e.g. "htf_itr_high", "htf_itr_eqh"
    direction: str | None       # "bullish" | "bearish"
    price: float | None         # level price
    seen_at: str | None         # confirmed_at or taken_at
    variant: str | None         # "level" | "eq" | None
    side: str | None            # "high" | "low" | None

02 HTF B-initiation Chain

The B-initiation pattern is the most complex: it is a three-step multi-bar accumulator. Unlike gate patterns that evaluate fresh each bar, this chain carries state from bar to bar. The candidate ID is stable across bars until an epoch boundary resets the accumulator.

Mental model: the market is setting up Phase B (the trade entry zone) when it grabs a source liquidity pool in the P/D direction, then grabs the opposite ITR level, and finally enters a counter-direction HTF SD zone (the "decision zone" where institutional orders are likely waiting).

Discretionary → Algorithm mapping

Step 1 · Source ITR grab — P/D-direction liquidity taken _observe_b_init_source_grab()

For a bullish market (direction == "long"), the source grab is a bullish grab-reclaim of an HTF ITR low or equal-low (htf_itr_low / htf_itr_eql). For bearish, it's a bearish grab of HTF ITR high or equal-high.

The EC checks liquidity["htf_itr_grab_reclaim_ready"] and the direction/source keys. When matched, an _ITRGrabRef is stored in _BInitiationState.source_itr_grab. The grab ref is idempotent — once step 1 fires, re-checks are ignored. 

direction == "long"  → expected grab: bullish, source in {htf_itr_low, htf_itr_eql}
direction == "short" → expected grab: bearish, source in {htf_itr_high, htf_itr_eqh}
Step 2 · Opposite ITR grab — counter-direction liquidity taken _observe_b_init_opposite_grab()

After step 1 fires, the market must also take the opposite ITR level. For bullish: a bearish grab of HTF ITR high or equal-high. This confirms the sweep-and-reverse pattern — both sides of the range have been tagged.

Step 2 only activates when step 1 is complete. The opposite grab ref is stored in _BInitiationState.opposite_itr_grab.

Step 3 · Decision zone — counter-direction HTF SD zone entered _observe_b_init_decision_zones()

After steps 1 and 2, the EC checks whether price is currently inside any counter-direction HTF SD zone (in_zone == true). For bullish: a supply zone on the HTF timeframe. For bearish: a demand zone.

Zone IDs are accumulated into decision_zone_ids — a running list, not replaced bar-by-bar. The candidate transitions to ready when all three steps have fired. 

direction == "long"  → counter zone direction: "supply"
direction == "short" → counter zone direction: "demand"

Status: "collecting" (steps 1–2 done, zone not yet entered)
        "ready"      (all 3 steps complete)

The formal readiness condition for the B-initiation chain:

$$ \text{B-init ready} \iff \text{source\_grab} \;\wedge\; \text{opposite\_grab} \;\wedge\; |\text{decision\_zone\_ids}| > 0 $$
The B-initiation candidate ID is stable once step 1 fires. It persists until the epoch boundary resets. blocked_reasons carry exactly which steps are still missing so the DAG can track progress without re-reading raw snapshots.

03 HTF Counter-reaction Candidate

This neutral Layer 3 candidate describes when HTF expansion has stalled and counter evidence has appeared. It does not decide that Phase D is valid; Layer 4 later checks the origin and selects a node.

Three sub-conditions combine: the HTF P/D stopped expanding (no new extreme this bar), an opposing SD zone was tapped or resolved, and a counter-direction LTF SD zone has been created. A special variant fires even on a new extreme if LTF bias has already flipped counter. The target liquidity-grab variant does not fire from a matching grab alone. It qualifies a current HTF P/D or relevant HTF EQ grab/reclaim against reclaim location, a fresh post-reclaim LTF ChoCh, and the resolving HTF close.

Discretionary → Algorithm mapping

Normal path — stopped expansion + opposing reaction + LTF counter SD _compile_htf_counter_reaction()

htf_pd_stopped_expanding: the current bar did not set a new running extreme (tracked by active_phase_e_extreme_price).

htf_opposing_sd_reaction: any of — HTF pullback confirmed by SD zone, counter HTF zone currently in_zone, or last resolved zone was counter-direction.

ltf_counter_sd_created: at least one counter-direction LTF SD zone exists.

Special path — new HTF extreme + LTF already counter-biased _compile_htf_counter_reaction()

Even when the HTF is still making new extremes, if the LTF bias is already running counter to the HTF direction, the counter-reaction candidate fires as a warning signal. This catches early-rotation setups before the HTF stops expanding.

Liquidity-grab variant — current reclaim plus post-reclaim ChoCh target contract

EC reads every currently triggerable HTF P/D and HTF EQ grab/reclaim event. It derives relation to the active HTF expansion extreme, checks that price reclaimed inside the active HTF P/D, and orders the latest confirmed LTF ChoCh after reclaimed_at. It keeps qualified events in liquidity_reclaim_candidates[]; EC does not select P/D before EQ and does not name a Layer 4 D node.

  liquidity event belongs to current triggerable scope
  P/D event belongs to active HTF P/D epoch
  EQ event is pro-side and near active HTF expansion extreme/current external range

pro-attempt chronology:
  fresh pro-HTF continuation attempt starts after the reaction/pullback origin
  pro attempt reaches this liquidity event before reclaim
  pro attempt fails to establish continuation

  relation = at_active_extreme | near_active_extreme |
             pro_side_external_range | unrelated | unknown
  near distance <= max(event pool tolerance, current HTF EQ tolerance)

post-reclaim counter ChoCh:
  ltf.last_sc.choch == true
  ltf.last_sc.breakDirection is counter to HTF direction
  ltf.last_sc.eventTimestamp >= liquidity.reclaimed_at

  current relevant grab/reclaim exists
  but post-reclaim ChoCh or HTF close resolution is incomplete

A matching grab alone must remain collecting. Clean counter Orderflow is not required: one confirmed counter-direction LTF ChoCh after reclaim is the intended structural confirmation. Layer 4 later checks valid origins, selects a D subtype, and consumes the stable liquidity_event_id.
Layer 3 never receives Phase E state back from Layer 4. The target neutral tracker is htf_expansion_context with htf_expansion_extreme_price, htf_expansion_extreme_at, and htf_expansion_epoch_id. Current phase_e_context and active_phase_e_extreme_* names are compatibility aliases pending runtime migration.

04 LTF Counter-story Gate

The C-phase gate. Fires when the LTF bias has fully flipped counter to the HTF direction. This represents the lower-timeframe building its own story against the higher-timeframe trend — an inducement / counter-move that the B-phase entry will eventually exploit.

The gate requires only one condition: ltf_struct.bias is counter to HTF direction. The candidate is ready when a counter LTF SD zone also exists (the POI to trade from). If no zone exists yet, it emits as collecting with blocked_reasons: ["no_ltf_counter_sd_zone_available"].

Discretionary → Algorithm mapping

POI selection — oldest counter-direction LTF SD zone, prefer returned _compile_ltf_counter_story()

LTF counter zones are sorted by created_at ascending. Zones with in_zone == true (price currently inside) are preferred as the selected POI. If none are in-zone, the oldest available zone is selected.

The location_context carries selected_poi_id and selected_poi_touched so the DAG can see which zone is the target and whether it is currently being tested. 

selected_poi = returned_zones[0] if returned_zones else ltf_counter_zones[0]
location_context["selected_poi_touched"] = bool(selected_poi and selected_poi["in_zone"])
The LTF counter-story candidate is always emitted fresh per bar (new uuid). Status is ready when both LTF bias is counter and at least one counter SD zone exists. The DAG decides whether C-phase is active or whether the story resolves without a trade.

05 HTF B-phase Setup Gate

Direct B-phase entry gate. Fires when the LTF has turned back toward the HTF direction with a pro-trend LTF SD zone available inside the HTF P/D pullback zone. This is the simpler entry path — no three-step chain needed, just structural alignment.

Two sub-types based on depth of the HTF pullback:

  • strict — HTF P/D value percent < 50%. Price is deep in the pullback, close to the source zone. Higher conviction entry.
  • shallow — HTF P/D value percent between 50% and 70%. Shallower retrace, acceptable only with additional pullback confirmation.

Discretionary → Algorithm mapping

LTF turned back — LTF bias matches HTF bias _compile_htf_b_phase_setup()

ltf_struct.bias must match the HTF bias direction. For bullish HTF: LTF bias must be bullish. This confirms the lower-timeframe has reversed its counter-move and is now aligned with the trend.

HTF pro-SD reaction — HTF demand/supply tapped or resolved _compile_htf_b_phase_setup()

Either the current price is inside a pro-direction HTF SD zone (in_zone == true) or the last resolved HTF zone was pro-direction. This provides evidence that the HTF level is reacting to the pullback zone.

strict_ready: location is strict AND HTF pro-SD is reacting. shallow_ready: location is shallow AND HTF pro-SD is reacting AND pullback confirmed.

The B-phase setup candidate only emits when the base_candidate condition is met — pullback context is present, location is in the valid P/D half, LTF turned back, and an LTF pro-SD zone exists. Below this threshold the pattern is silent, keeping the candidate list clean.

06 HTF P/D Objective Gate

The A-phase finale gate. Fires when the current HTF bar touches the P/D range extreme — the objective price level where the trend is expected to reach its target. This is the simplest pattern: purely price-level based, requires HTF bar data.

For bullish direction, the objective is htf_struct["range_high"]. The bar high must reach or exceed it. The sub-flag phase_a_finale_closed_beyond is true when the HTF bar close is past the objective — indicating a breakout rather than a test.

$$ \text{touched}_\text{long} = \text{bar.high} \ge \text{range\_high} $$ $$ \text{closed\_beyond}_\text{long} = \text{bar.close} > \text{range\_high} $$
htf_pd_objective only emits when the bar actually touches the objective — no suppression logic, no partial state. If no HTF bars are passed to update() or the structure has no range_high / range_low, the pattern is silent.

07 Phase-E Context

The always-on context emitter. Every bar when HTF direction is known, this pattern fires with status ready. It gives the DAG a running summary of the current E-cycle state without the DAG needing to read HTF bars directly.

Three signals packed into debug_facts:

  • Extreme tracker — running high (long) or low (short) of the current P/D expansion: active_phase_e_extreme_price, active_phase_e_extreme_time.
  • Two-bar reaction signal — compares previous HTF bar vs current bar: reaction_confirmed (bar closed past the prior bar's extreme) and reaction_warning (wick crossed but close reversed). Used by DAG to detect D-phase inception.
  • Reaction-failed flag — reaction_failed is true when the current bar close is past the pre-update extreme watermark. Detects D→E failure: price swept the prior extreme and kept going.

Pre-update extreme ordering

A subtle ordering issue: _update_phase_e_extreme() is called before _compile_phase_e_context(), so if both run in the same bar, the extreme is already updated to include the current bar's movement when the context compiler runs. This would cause reaction_failed to compare the close against the current bar's new watermark rather than the previous watermark — always false.

Fix: the extreme is snapped before the update call and passed as pre_update_extreme: 

pre_update_extreme = self.active_phase_e_extreme_price      # save before
self._update_phase_e_extreme(fused, htf_struct, direction, htf_bar)
# ...
phase_e = self._compile_phase_e_context(..., pre_update_extreme=pre_update_extreme)
phase_e_context is the DAG's heartbeat signal for the expansion cycle. It is emitted every bar regardless of whether the other patterns fire. The DAG should read it first to understand the current E-phase state, then interpret the other candidates against it.

08 Algorithm

EvidenceCompiler.update() is called once per bar from dual_smc.py. It returns the full candidate list — one object per active pattern. Callers store the result in payload["evidence_candidates"]. 

# EvidenceCompiler.update() — called once per bar
def update(self, fused, *, higher_bars=None) -> list[EvidenceCandidate]:
    htf_struct  = _htf_struct(fused)
    direction   = _direction_from_bias(_htf_ctx(fused).get("bias"))
    timestamp   = _current_timestamp(fused)

    # 1. Epoch boundary — reset stateful accumulators
    epoch_id = _epoch_id(htf_struct)
    if epoch_id and epoch_id != self.htf_pd_epoch_id:
        self.htf_pd_epoch_id = epoch_id
        self._b_init.reset()
        self.active_phase_e_direction  = "none"
        self.active_phase_e_extreme_price = None

    htf_bar, htf_prev_bar = _htf_bars_current_and_previous(higher_bars)
    if direction == "none" or htf_struct is None:
        return []

    # 2. Save extreme BEFORE updating (for reaction_failed)
    pre_update_extreme = self.active_phase_e_extreme_price
    self._update_phase_e_extreme(fused, htf_struct, direction, htf_bar)

    candidates = []
    # 3. Run all six pattern compilers
    for fn in [
        lambda: self._compile_b_initiation(fused, direction, timestamp),
        lambda: self._compile_htf_counter_reaction(fused, htf_struct, direction, htf_bar, timestamp),
        lambda: self._compile_ltf_counter_story(fused, direction, timestamp),
        lambda: self._compile_htf_b_phase_setup(fused, htf_struct, direction, timestamp),
        lambda: self._compile_htf_pd_objective(fused, htf_struct, direction, htf_bar, timestamp),
        lambda: self._compile_phase_e_context(fused, htf_struct, direction,
                    htf_bar, htf_prev_bar, timestamp,
                    pre_update_extreme=pre_update_extreme),
    ]:
        result = fn()
        if result is not None:
            candidates.append(result)
    return candidates
# Multi-bar accumulator — stateful across bars via _BInitiationState
def _compile_b_initiation(self, fused, direction, timestamp):
    liquidity = _htf_liquidity(fused)
    htf_label = _reference_tf(fused)

    # Try to advance each step (idempotent once a step fires)
    self._observe_b_init_source_grab(liquidity, direction, timestamp)
    self._observe_b_init_opposite_grab(liquidity, direction, timestamp)
    current_decision_zones = self._observe_b_init_decision_zones(fused, direction, htf_label)

    b = self._b_init
    if not b.source_itr_grab and not b.opposite_itr_grab and not b.decision_zone_ids:
        return None  # Nothing yet — suppress until step 1 fires

    status = "ready" if b.ready else "collecting"

    return EvidenceCandidate(
        candidate_id = b.candidate_id,  # STABLE per epoch
        pattern      = "htf_b_initiation",
        status       = status,
        direction    = direction,
        timeframe    = htf_label,
        evidence_refs = [
            r.to_evidence_ref()
            for r in [b.source_itr_grab, b.opposite_itr_grab]
            if r is not None
        ],
        source_object_refs = [
            {"kind": "sd_zone", "zone_id": zid, "timeframe": htf_label}
            for zid in b.decision_zone_ids
        ],
        blocked_reasons = [] if b.ready else [
            s for s in [
                None if b.source_itr_grab else "source_itr_grab_not_seen",
                None if (not b.source_itr_grab or b.opposite_itr_grab) else "opposite_itr_grab_not_seen",
                None if (not (b.source_itr_grab and b.opposite_itr_grab) or b.decision_zone_ids) else "decision_zone_not_seen",
            ] if s is not None
        ],
        ...
    )
# Phase-E emitted every bar when direction is known
def _compile_phase_e_context(self, fused, htf_struct, direction,
                               htf_bar, htf_prev_bar, timestamp,
                               *, pre_update_extreme=None):
    # Two-bar reaction signal
    reaction_confirmed = reaction_warning = False
    if htf_bar and htf_prev_bar:
        if direction == "long":
            prev_low  = float(htf_prev_bar["low"])
            cur_low   = float(htf_bar["low"])
            cur_close = float(htf_bar["close"])
            # Bearish bar breaks below previous low — D-phase inception signal
            reaction_confirmed = cur_low < prev_low and cur_close < prev_low
            reaction_warning   = cur_low < prev_low and cur_close >= prev_low

    # Reaction-failed against PRE-UPDATE watermark
    reaction_failed = False
    if htf_bar and pre_update_extreme is not None:
        close = float(htf_bar["close"])
        if direction == "long":
            reaction_failed = close > pre_update_extreme
            # close above previous E-extreme → D-phase failed, trend resumed

    return EvidenceCandidate(
        pattern  = "phase_e_context",
        status   = "ready",
        debug_facts = {
            "active_phase_e_extreme_price": self.active_phase_e_extreme_price,
            "new_htf_extreme":    self._new_phase_e_extreme(direction, htf_bar),
            "reaction_confirmed": reaction_confirmed,
            "reaction_warning":   reaction_warning,
            "reaction_failed":   reaction_failed,
            ...
        }, ...
    )

09 Open Issues · Edge Cases · Next Milestone

Edge Case

htf_itr_grab_reclaim key collision in liquidity_variant. The _htf_counter_reaction_liquidity_variant() dict spread used to overwrite the locally computed "trigger" key with a potentially-None value from the spread. Fixed: **liq_grab is now spread first in debug_facts so named keys take precedence. Verify if new liquidity keys are added to the passthrough dict.

Edge Case

higher_bars not always available. If the caller omits higher_bars, htf_pd_objective silently returns None and phase_e_context emits with no reaction signal. The replayer and dual_smc must ensure at least 2 recent HTF bars are always passed. Currently not enforced at the EC boundary.

Out of Scope

Orderflow integration into candidates. The EC does not yet consume an OrderflowSnapshot. The 301-context design says EC should accept a projected orderflow read as soft support for candidate tilting. Deferred until Orderflow Context (305) is production-ready and consistently populated.

Out of Scope

Candidate expiry and invalidation. Status values "invalidated" and "expired" exist in the type alias but are not emitted by any pattern today. The DAG is expected to track expiry itself. Add EC-driven expiry once the DAG's attention window logic is defined.

Fallback

Epoch boundary too sensitive. If last_sc timestamps vary slightly between bars (e.g. rounding), the epoch ID may flip every bar, resetting B-initiation on every tick. If B-init never accumulates, check that htf_struct["last_sc"]["eventTimestamp"] is stable across bars for the same structural event. Use the replayer's debug log to inspect epoch_id drift.
