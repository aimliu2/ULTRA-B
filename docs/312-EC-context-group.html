Layer 3 → Layer 4 · Evidence Compiler

EC Candidates — Grouped by Phase

TL;DR — The Evidence Compiler emits 7 candidate patterns per bar. This page groups them by the DAG phase they feed so you can look up "what does Phase X receive from EC?" rather than reading the technical groups in 311. Each section names the pattern, the consuming hypothesis.py method, and all debug_facts keys with their role in the phase transition logic.
→ 311 EC Context — technical reference → 401 Hypothesis DAG → 402 Phase D model

00 Overview

Each row below is one EC candidate pattern. The right column names every hypothesis.py method that calls _ec_candidate(snapshot, "…") on it. ltf_counter_choch is the only pattern consumed by two phases.

01 Phase E

E Primary candidate: htf_expansion_context (always ready when direction known)
Compat alias: phase_e_context — identical copy, emitted right after _compile_htf_expansion_context()

Emitted every bar when HTF direction is known. Status is always ready. The candidate carries 51 debug_facts that feed the Phase E four-node machine (E.seeking → E.HTF_reaction | E.stalling → E.pullback_developing). _phase_e_reaction() reads it and re-exposes every key prefixed phase_e_context_ in the hypothesis debug dict, which _phase_e_shadow_facts() then reads to drive node transitions.

02 Phase D

D Target gate reads: phase_e_context provenance + ltf_counter_choch
Supporting quality/liquidity: htf_counter_reaction D.watch / D.speculation state machine

The target D.watch gate is lineage-aware: Layer 4 reads the orderflow MSS provenance from phase_e_context, the structural event provenance from ltf_counter_choch, and Phase E shadow's post-E pro-attempt memory. htf_counter_reaction remains useful as supporting quality/liquidity evidence and for archived liquidity-test paths. Layer 4 owns D-node selection — EC emits no phase_d_* keys.

ltf_counter_choch — structural event stream for Phase D

Layer 4 reads _ec_candidate(snapshot, "ltf_counter_choch") for structural event chronology. The target D.watch gate also requires event provenance and Phase E pro-attempt separation; a fresh ChoCh alone is no longer enough to open D.watch.

Layer 4 owns D-node selection. _phase_d_setup() derives liq_selected_node from liquidity_reclaim_candidates + liquidity_reclaim_ready_event_ids — EC emits no phase_d_* keys.

03 Phase C

C Candidate: ltf_counter_story _phase_c_setup()

The C-phase gate. Fires when the LTF bias has fully flipped counter to the HTF direction. Status ready when LTF bias is counter and at least one counter LTF SD zone exists (the POI to trade from). Otherwise emits as collecting with blocked_reasons: ["no_ltf_counter_sd_zone_available"].

ltf_counter_story facts _compile_ltf_counter_story()
ltf_bias_counter_htf bool LTF structural bias counter to HTF — the gate condition ltf_counter_pd_break bool LTF broke through its P/D range counter-HTF ltf_counter_break_direction str|None "down" (long) | "up" (short) ltf_counter_pullback_confirmed bool LTF pullback structure confirmed ltf_counter_bos_confirmed bool LTF BoS confirmed in counter direction (vote-based) ltf_counter_sd_zone_ids list all available counter LTF SD zone IDs ltf_counter_sd_returned_zone_ids list returned (in_zone == True) counter LTF SD zones selected_poi_id str|None zone_id of the preferred POI selected_poi dict|None full zone dict of the selected POI

POI selection: counter zones sorted by created_at ascending. Returned zones preferred; oldest available zone used as fallback. _phase_c_setup() reads selected_poi.in_zone to set phase_c_selected_poi_touched.

04 Phase B

Phase B has two separate EC patterns for two entry paths: a direct setup gate and a stateful three-step initiation chain.

B Pattern 1: htf_b_phase_setup — direct entry gate _phase_b_setup()

Fires when LTF has turned back toward HTF direction with a pro-trend LTF SD zone inside the valid HTF P/D pullback half. Silent (returns None) unless the base candidate condition is met — keeping the candidate list clean. Two sub-types by depth: strict (< 50% of P/D span) and shallow (50–70%).

B Pattern 2: htf_b_initiation — 3-step multi-bar accumulator _phase_b_initiation_setup()

$$ \text{B-init ready} \iff \text{source\_grab} \;\wedge\; \text{opposite\_grab} \;\wedge\; |\text{decision\_zone\_ids}| > 0 $$
Step directions: for a long market, step 1 is a bullish grab of htf_itr_low or htf_itr_eql; step 2 is a bearish grab of htf_itr_high or htf_itr_eqh; step 3 is entering an HTF supply zone. Reversed for short. Steps are idempotent — once a step fires it stays fired until epoch reset.

05 Phase A

A Candidate: htf_pd_objective _phase_a_finale()

The A-phase finale gate. The simplest pattern: purely price-level based. Silent (returns None) unless the current HTF bar actually touches the P/D objective. No partial state, no collecting status. The phase_a_finale_closed_beyond sub-flag distinguishes a breakout from a test.

htf_pd_objective facts _compile_htf_pd_objective()
phase_a_finale_touched bool bar.high ≥ range_high (long) | bar.low ≤ range_low (short) phase_a_finale_closed_beyond bool bar.close past objective — breakout vs test phase_a_finale_direction str "long" | "short" phase_a_finale_objective float the objective price level (range_high or range_low) phase_a_finale_rule str "bullish_phase_a_close_above_htf_pd_objective" "bullish_phase_a_touch_without_close_above_htf_pd_objective" "bearish_phase_a_close_below_htf_pd_objective" "bearish_phase_a_touch_without_close_below_htf_pd_objective" current_htf_high float bar high current_htf_low float bar low current_htf_close float bar close
$$ \text{touched}_\text{long} = \text{bar.high} \ge \text{range\_high} \qquad \text{closed\_beyond}_\text{long} = \text{bar.close} > \text{range\_high} $$

06 Open Issues

Next Milestone

Phase D new model (D.watch / D.speculation) not yet coded. Design settled in AGENTS.md. _phase_d_setup() still uses the old 4-variant model. PhaseDShadow dataclass, the ltf_itr_level_run EC fact, and same-bar ChoCh guard must be added before the new transitions can be wired.

Next Milestone

Phase C new model (C.pullback / C.pullback_weaken) not yet coded. Old sub-statuses superseded by design. Phase C gates are currently commented out in hypothesis.py (~line 436 and ~line 456). Re-enable after D model is confirmed and working.

Next Milestone

Phase A setup candidate missing. _phase_a_setup() still reads flat LTF/pro-zone snapshot fields directly. EC does not emit a dedicated Phase A setup candidate. Until it does, HypothesisClassifier.classify() signature cannot be narrowed to a pure candidate list.

Open Issue · Chronology

Same-timestamp ChoCh guard for D.watch. The new D model requires a ChoCh after E.pullback_developing. Using >= on eventTimestamp allows the same LTF bar that triggered the pullback to also provide the confirming ChoCh. Strict > or an event-id guard may be needed.

Next Milestone

classify() signature narrowing. Once all phase methods read EC candidates, the public interface of HypothesisClassifier.classify() should accept evidence_candidates: list[dict] instead of the raw snapshot dict, severing the direct Layer 3 dependency from Layer 4.
