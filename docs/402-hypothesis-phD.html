Layer 4 · Hypothesis Classifier · Phase D

Phase D — Reaction Boundary

TL;DR - Phase D is the first global reaction phase after Phase E expansion stops being simple continuation. The settled paths are reaction-point and HTF liquidity grab/reclaim tests. The unsettled path is the second pullback after E.pullback_developing breaks and returns to E.stalling; that may deserve a D node, but its name and trigger are still open.
→ Back to 401 Hypothesis Dataflow → Open 401 Hypothesis DAG → Related: Phase E Detail

00 Core Concept

Phase D is the boundary where the system stops treating a counter move as soft Phase E memory and starts treating it as a global reaction process. It is still pre-entry. Layer 5 should not trade D by itself; D exists to name the reaction and prepare the later C pullback path.

The important separation is that E.pullback_developing can be a first soft pullback inside Phase E, but repeated counter movement after a failed pro-continuation attempt should not loop forever inside E. That second pullback is the unresolved Phase D naming question.

01 Current Phase D

As coded now, Phase D is selected from the EC candidate htf_counter_reaction. Layer 4 reads the candidate, keeps the debug keys, and maps it into the current D sub-status.

Reaction-point D EC → DAG

The generic D path is D.reaction_point. It means HTF Phase E has stopped clean expansion and enough counter-reaction evidence exists to stop calling the move a simple E stall. 

EC:
  pattern = htf_counter_reaction
  status  = ready
  facts:
    htf_pd_stopped_expanding
    htf_opposing_sd_reaction
    ltf_bias_counter_htf

DAG:
  if previous/current E and phase_d_ready:
    phase = D
    phase_sub_status = reaction_point
Liquidity-test D variants HTF liquidity + LTF structure → EC → DAG

D.htf_pd_grab_reclaim and D.htf_eq_grab_reclaim are the two liquidity-test paths. Each requires a full chronological sequence confirmed by two Layer 3 inputs: the HTF liquidity event and the LTF structure change-of-character (choch).

Phase D is not an entry phase. It names a reaction boundary so Phase C can later decide whether the pullback story is mature enough to arm or watch a counter POI.

02 Failed Pro-Attempt Boundary

The unresolved case starts inside Phase E. A first clean LTF counter orderflow leg moves E.stalling into E.pullback_developing. If that counter leg is disrupted and a pro-continuation test appears without a new HTF extreme, the shadow memory may return to E.stalling. If price then pulls back again, that should probably graduate to Phase D instead of creating an endless E loop.

Discretionary read chart logic

The first counter leg can still be treated as cooling inside Phase E. But after it breaks and the pro-side attempt fails to expand the HTF P/D range, the next counter leg is no longer just a first pullback. It is evidence that expansion is reacting. 

First pullback:
  E.seeking → E.stalling → E.pullback_developing

Pullback disrupted:
  E.pullback_developing → E.stalling

Second pullback after failed pro-continuation:
  E.stalling → D.?   # unresolved name
Proposed component ownership EC · DAG · Shadow Thesis 
EC owns:
  ltf_counter_orderflow_clean
  ltf_counter_orderflow_broken
  ltf_counter_orderflow_leg_id
  new_htf_extreme
  reaction facts if present

Shadow Thesis / phase_e owns:
  first_counter_leg_id
  pro_continuation_leg_id
  failure_counter_leg_id
  whether the same stale leg was already consumed

DAG owns:
  first clean counter pullback stays E.pullback_developing
  disrupted pullback can return to E.stalling
  second clean counter pullback after disruption may select D.?
Do not let Phase E absorb unlimited failed continuation attempts. One soft pullback can remain E memory; repeated pullback after disruption should be considered for a D boundary.

05 Open Issues

Open Issue

Name the second-pullback D node. Current proposed label: D.continuation_failure_test. Earlier alternatives included D.pro_attempt_failed, D.failed_pro_attempt, and D.fail.proattempt. The unresolved meaning is: E.pullback_developing was disrupted, returned to E.stalling, then a new clean counter pullback appeared without HTF expansion. Confirm that this is a separate D subtype rather than a reaction-point variant before coding it.

Open Issue

Prevent the E pullback loop from hiding a D reaction. Once Phase E has already seen a clean counter pullback and that pullback was disrupted, the next clean counter pullback should not simply repeat E.stalling → E.pullback_developing. The DAG needs a memory-aware gate that can graduate the second pullback into Phase D.

Open Issue

Move Phase E shadow memory under root Shadow Thesis. Current code keeps state.phase_e_shadow beside state.shadow_thesis, while the architecture note now defines Shadow Thesis as the root Layer 4 memory component with phase-owned ledgers. The implementation should eventually become state.shadow_thesis.phase_e while preserving emitted debug keys.

Edge Case

Existing D names are liquidity/reaction oriented. Current sub-statuses are reaction_point, htf_pd_grab_reclaim_test, and htf_eq_grab_reclaim_test. A failed-pro-attempt D node may need different wording so it does not imply a confirmed HTF liquidity grab or opposing S/D reaction.

Current Behavior Conflict

Revisit new_htf_extreme_with_ltf_counter_bias as a D trigger. The current htf_counter_reaction candidate can become ready when price makes a strict new HTF extreme while LTF bias is counter. Under the proposed failed-continuation definition, a strict new HTF extreme proves that the resumed pro-direction continuation succeeded and should normally return or hold E.seeking. D should require a separate explicit liquidity or reaction footprint on that path.

Current Behavior Conflict

Do not reuse reaction_failed for failed continuation. The current EC fact means that an existing D/C reaction collapsed because price closed through the remembered Phase E extreme and resumed the original HTF direction. The new failed-continuation D origin means the opposite chronology: the pro-direction attempt failed before creating a strict new HTF extreme, then a fresh counter leg appeared. These facts require separate names and separate transition guards.

Partially Resolved · Liquidity Lifecycle

Consume each liquidity grab/reclaim event exactly once. liquidity_event_id (pool_id + confirmed_at) and the htf_pd_epoch_id epoch match are now in the EC contract. Layer 4 still has no consumed-event memory — phase_d_liquidity_test_status stays at pending_outcome and a failed D reaction can immediately reopen the same event. Next: add a Shadow Thesis ledger entry that marks an event consumed when D confirms into C, fails into E, or expires; require a freshly confirmed event before reopening the same subtype.

Partially Resolved · Liquidity Scope

Prevent stale P/D and EQ liquidity from opening Phase D. EC now gates each event with scope in {active_current_epoch, carryover_recent} and htf_pd_epoch_id match. Old P/D pools from a prior epoch no longer qualify. Remaining gap: EQ pool rescoping on P/D epoch changes and expiry by HTF age are not yet enforced. relation_to_active_phase_e_extreme filtering for EQ is in place but the external_archive purge path is still open.

Open Issue · Candidate Qualification

Choose liquidity D candidates by current relevance, not fixed availability. EC currently accepts any ready P/D or EQ grab whose direction counters HTF bias. It does not require an EQ pool to relate to the active HTF expansion extreme or current external range, and if P/D and EQ candidates are ready together, P/D always wins because it is appended first. Define relation-to-current-E facts, recency, and relevance ranking before Layer 4 selects D.htf_pd_grab_reclaim_test or D.htf_eq_grab_reclaim_test.

Open Issue · Confirmation Strength

Decide whether one LTF candle is sufficient to open a liquidity D test. The current liquidity lifecycle allows one candle to breach the HTF level, close back inside it, and provide its own rejection wick or hammer-like evidence. That immediately confirms the grab/reclaim and can open D. Confirm whether same-bar sweep/reclaim is the intended minimum evidence, or require a later reclaim/follow-through bar or independent counter evidence before exposing a D-ready event.

Open Issue · Choch as Separate EC Stream

Promote LTF counter choch to a standalone top-level EC candidate. ltf_counter_choch_after_reclaim is already a hard required gate inside the per-candidate dict of the htf_counter_reaction liquidity variant, but it is not emitted as a top-level EC pattern. Layer 4 currently cannot read the choch fact independently of the full liquidity sequence. Add a _compile_ltf_counter_choch() compile step that emits an EvidenceCandidate with pattern ltf_counter_choch and facts ltf_counter_choch_seen, ltf_counter_choch_event_at, ltf_counter_choch_direction, ltf_counter_choch_level. Layer 4 then reads it via _ec_candidate(snapshot, "ltf_counter_choch") in addition to the existing htf_counter_reaction gate.
