Hypothesis none — Warmup & Unclassified Bars

TL;DR — phase = "none" is not an error. It is the honest label for every bar where no E/D/C/B gate fires. It carries a none_sub_status that distinguishes structural warmup, HTF bias absence, the 15m/4H first-bar silence, and genuinely unclassified market conditions. The warmup loop feeds structure and liquidity engines only — the hypothesis classifier never runs during warmup. After warmup, the first real hypothesis is always Phase E; Phase D requires Phase E to have been committed at least once.

00 Core Concept — none is a first-class phase

The classifier is called on every bar. When no phase gate fires — no D evidence, no C story, no B POI, no A alignment, no E open leg — the classifier emits phase = "none". This is explicit, not a fallback. It is the correct output for bars where the market has not yet shown enough information to commit to a directional thesis.

none carries a none_sub_status string that explains why the bar was unclassifiable. Measuring none frequency and sub_status distribution is a primary debug signal — a classifier that never emits none is miscalibrated.

none_sub_status enum

All values found in hypothesis.py:

none_sub_status When it fires Source location
warmup_waiting_for_first_closed_htf No HTF structure at all (not htf) but cursor_time exists, one HTF bar exists, no previous bar, and the bar time is ahead of cursor_time. Cold-start signal for the 15m/4H combo. classify() top branch — no-htf path and htf_neutral path
waiting_for_htf_structure No HTF structure snapshot at all (htf is None). Structure engine has not yet emitted a valid bias. Covers the structural warmup period. classify() — not htf branch
htf_neutral HTF structure exists but bias == "neutral" or direction resolves to "none". Market is in range or between structural epochs. classify() — bias == "neutral" branch
htf_resolve_unclassified HTF has a directional bias and a valid phase, but no E/D/C/B gate fired. The phase is open or pullback_confirmed but none of the slice conditions matched. Usually means the classifier's slice did not handle this sub-state. classify() — final fallback
waiting_for_hypothesis_state _carry_current_hypothesis() was called but current_hypothesis is None. Internal guard. _carry_current_hypothesis()
Record none honestly. Do not suppress it, roll it into the last active phase, or treat it as a missing-data artifact. The none_sub_status distribution reveals which structural states the classifier cannot yet handle — it is a roadmap for the next development slice.

01 Warmup Architecture

Before the replay loop starts, dual_smc._init_engines() calls two warmup methods that feed historical bars to the structure and liquidity engines. This is entirely separate from the hypothesis classifier.

What warmup feeds

_warm_lower_engines() — LTF structure and zone engines dual_smc.py

Feeds lower_bars[lower_start_index - warmup_bars : lower_start_index] to the LTF StructureEngine and zone engines via _process_lower_engines(row). No snapshot() call. No hypothesis output.

_warm_higher_engines() — HTF structure engine dual_smc.py

Feeds higher_bars[higher_start_index - warmup_bars : higher_start_index] to the HTF StructureEngine via _process_higher_engines(row). After this loop, current_higher_index = higher_start_index - 1 — pointing to the last warmed-up bar, ready for the replay to advance from there.

_warm_liquidity_lower_interactions() — liquidity context dual_smc.py

Feeds LTF bars to LiquidityContextEngine.on_lower_bar() for the warmup window. Populates ITR pools, grab states, and reclaim readiness before bar 0 of the replay.

What warmup does NOT feed

  • HypothesisClassifier — never called during warmup. State is at init defaults: previous_phase = "none" and active_phase_e_direction = "none".
  • EvidenceCompiler — never called during warmup. _BInitiationState is fresh. No epoch ID set.
  • Fusion / Normalizer / Interpreter — not called during warmup bar feed.
After warmup: each enabled timeframe's Layer 3 engines have independently consumed up to 200 bars, so HTF bias and phase are typically established at replay bar 0. The hypothesis classifier and EC start completely fresh. The classifier's first output is always based on the state produced by warmup, never on warmup bars themselves.

02 The 15m / 4H Combo Scenario

In a 15m execution / 4H context configuration, one closed 4H candle = 16 closed 15m bars. The hypothesis classifier needs at least two closed 4H bars before the two-bar reaction signal can produce any output. This creates a cold-start silence window.

_waiting_for_first_closed_htf() — the guard

Called at the top of classify() on every bar. Returns True when all four conditions hold: 

cursor_time     is not None        # current 15m bar time is known
current_bar     is not None        # at least one 4H bar exists
previous_bar    is None            # only one closed 4H bar seen so far
current_bar["time"] > cursor_time  # the 4H bar time is ahead of cursor_time

When this fires, none_sub_status is set to "warmup_waiting_for_first_closed_htf" in both the not htf branch and the htf_neutral branch of classify().

Cold-start silence window

If the replay starts with no prior 4H history (no warmup, or insufficient warmup):

bars 0–15
no 4H close
current_bar=None
›
bars 16–31
1 closed 4H
previous=None
→ warmup_waiting
›
bar 32+
2 closed 4H
reaction signal
available

During bars 0–15 the HTF structure snapshot has no bars at all — the classifier emits none / waiting_for_htf_structure. During bars 16–31, previous_bar is None and the warmup guard fires. From bar 32 onward, two closed 4H bars exist and the full reaction signal is available.

The runtime applies warmup_bars: 200 independently: it feeds up to 200 closed 15m bars to the LTF engines and up to 200 closed 4H bars to the HTF engines. It does not derive the HTF warmup by resampling the 200 LTF bars. This normally ensures that by replay bar 0, the HTF structure engine already has multiple closed 4H bars and previous_bar is not None. The warmup_waiting_for_first_closed_htf sub_status should never appear in a properly configured replay.

03 Structure Engine Warmup — The Hidden Dependency

Even with 200 warmup bars fully fed to the structure engine, the engine itself has an internal data dependency that can produce a secondary silence window at the start of the replay. This is not the hypothesis classifier's warmup — it is the structure engine's own data requirement.

Structure-break + liquidity ATR dependencies

Structure initialization — enough pivots and a structural break StructureEngine

The structure engine has no fixed numeric warmup requirement. It needs enough price history to confirm STR pivots, compress them into the selected ITR/LTR tier, and then observe a close through a registered structural level. That break establishes the directional bias and opens the first P/D range.

Liquidity ATR(14) — the hard numeric floor LiquidityContextEngine

liquidity.eq.atr_period: 14 requires 14 closed HTF bars before EQH/EQL and ITR-equality tolerances are available. This ATR belongs to the liquidity engine; it does not establish StructureEngine bias. Fourteen bars is the hard configured indicator floor, not the reason for choosing a 200-bar warmup.

Sideways-market guardrail — why the configured warmup is 200 design decision

warmup_bars: 200 is an intentional guardrail for a market that may spend roughly a week moving sideways. A shorter warmup such as 100 bars will often satisfy ATR and initialize structure, but it is more likely to begin entirely inside that sideways regime and lose the directional structure that preceded it. The extra history gives Layer 3 a better chance to retain the earlier pivots, structural break, active P/D range, orderflow, and liquidity context needed to interpret the range. 

warmup_bars: 200 is applied independently to both sides of a dual combo:

  5m / 1H   → up to 200 × 5m bars  + 200 × 1H bars
  15m / 4H  → up to 200 × 15m bars + 200 × 4H bars
  1H / 1D   → up to 200 × 1H bars  + 200 × 1D bars

Hard numeric floor:
  liquidity EQH/EQL tolerance requires 14 closed HTF bars

Chosen operating guardrail:
  200 bars gives Layer 3 room to look through roughly a week of sideways price
  and recover useful pre-range structural context before Layer 4 starts
If you observe sustained none / htf_neutral output after replay start, the structure engine is the root cause — not the hypothesis classifier. Check htf["bias"] in the payload to confirm. The 200-bar guardrail reduces the chance that a week-long sideways regime hides all preceding directional structure. If the complete warmup still contains no usable structural break, the market is genuinely unresolved and the correct response is to change the instrument, not fight the config. none = no trade.

04 First Hypothesis After Warmup

When the structure engine finally produces a valid directional bias, the classifier can emit a real hypothesis. The first real hypothesis is always Phase E — never Phase D, C, or B directly.

Phase D guard — _previous_or_active_e()

Phase D entry in classify() is gated by: 

if (
    self.state.previous_phase not in {"B", "A"}
    and self._previous_or_active_e(direction)
    and phase_d["phase_d_ready"]
):
    hyp = self._phase_d(...)

def _previous_or_active_e(self, direction: Direction) -> bool:
    return (
        self.state.previous_phase == "E"
        or self.state.active_phase_e_direction == direction
    )

On the first bar after warmup, HypothesisClassifierState is at init defaults: 

previous_phase           = "none"    # init default
active_phase_e_direction = "none"    # init default

Both conditions in _previous_or_active_e() are False → the normal Phase D gate does not evaluate. Similar origin guards affect later phase paths. As coded now, the only reliable self-location path from warmup is the fallback Phase E path when HTF phase is open.

Discretionary → Algorithm mapping

Bar 0 — first valid bias bar classify() open-phase fallback

HTF has directional bias, phase is open, no D/C/B gate fires (because _previous_or_active_e() is False). The classifier falls through to: 

if phase == "open":
    hyp = self._phase_e(direction, cursor_time, debug, ltf)
    return self._commit(hyp)

# After commit:
# previous_phase           → "E"
# active_phase_e_direction → direction

Phase E is emitted. Shadow node is set to E.seeking.

Bar 1+ — D and C now available classify() main gate

After Phase E is committed on bar 0, previous_phase == "E" and active_phase_e_direction == direction. Now _previous_or_active_e() returns True. Phase D, C, and B gates can evaluate from bar 1 onward.

The two-bar reaction signal also becomes available on bar 1 (assuming two closed HTF bars exist from warmup), so the reaction-warning and reaction-confirmed flags are meaningful from bar 1.

Target rule: the first post-warmup bar should self-locate from none(warmup) into the best current DAG node: Phase E, D, C, B, or A. The current implementation only handles the easy none(warmup) → E case.

05 Layer 3/4 Boundary Check

Per the .claude/layer34-contract.md analytic guide, the warmup and none period must not violate the three boundary contracts between the Evidence Compiler (EC), DAG (hypothesis classifier), and their shared output.

Contract 1 — EC → DAG during none period

The EC's update() returns an empty list when direction == "none" or htf_struct is None. This produces: 

payload["evidence_candidates"] = []     # valid — EC is silent
payload["evidence_compiler_epoch_id"] = None

The DAG reads candidates via: 

ec = {c["pattern"]: c for c in payload.get("evidence_candidates", [])}
# → ec = {} (empty dict)
b_init = ec.get("htf_b_initiation")   # → None
d_gate = ec.get("htf_counter_reaction") # → None
# all candidates are None → no gate fires → none emitted

Verdict: Contract 1 is safe. Empty candidates is a valid state. The DAG handles it gracefully via .get() defaulting to None.

Contract 2 (DAG → Shadow) is unaffected. Shadow state is not read or written during none emission — the shadow methods are only called from Phase E/B-specific paths, not from _none().
Contract 3 (Shadow → DAG output) is unaffected. payload["hypothesis"] is always emitted, even during none. The schema is the same — phase = "none", direction = "none", status = "watching". No field goes missing.

The one real risk

Downstream consumers (Layer 5, logging, dashboards) that assume evidence_candidates is non-empty after bar 0 will silently fail during the none period. The contract guarantees the key exists but not that it is non-empty. Any code that does: 

# ✗ WRONG — assumes non-empty
candidates[0]["pattern"]

# ✓ CORRECT — always safe
ec = {c["pattern"]: c for c in payload.get("evidence_candidates", [])}
b_init = ec.get("htf_b_initiation")
if b_init and b_init["status"] == "ready":
    ...

06 Algorithm

# classify() — warmup and none paths (simplified)
def classify(self, snapshot):
    htf          = snapshot.get("higher_structure") or snapshot.get("structure")
    cursor_time  = snapshot.get("cursor_time")
    current_bar, previous_bar = _current_and_previous_htf_bars(snapshot)

    waiting = _waiting_for_first_closed_htf(cursor_time, current_bar, previous_bar)

    # ── No HTF structure at all ──────────────────────────────
    if not htf:
        hyp = self._none(
            "waiting_for_htf_structure",
            ["htf_structure"],
            "HTF structure snapshot becomes available",
            cursor_time, debug,
            none_sub_status=(
                "warmup_waiting_for_first_closed_htf"
                if waiting else "waiting_for_htf_structure"
            ),
        )
        return self._commit(hyp)

    # ── HTF neutral or no bias ───────────────────────────────
    if bias == "neutral" or direction == "none":
        hyp = self._none(
            "HTF bias is neutral; no directional hypothesis",
            ["directional_htf_bias"],
            "HTF bias becomes directional",
            cursor_time, debug,
            none_sub_status=(
                "warmup_waiting_for_first_closed_htf"
                if waiting else "htf_neutral"
            ),
        )
        return self._commit(hyp)

    # ── No slice matched ─────────────────────────────────────
    hyp = self._none(
        "E/D/C/B classifier slice found no tradable hypothesis",
        ["phase_a_classifier"],
        "Next classifier slice defines this pullback/continuation state",
        cursor_time, debug,
        none_sub_status="htf_resolve_unclassified",
    )
    return self._commit(hyp)
# Guard for 15m/4H cold-start — only True when BOTH:
#   • one closed HTF bar exists
#   • no second closed HTF bar yet (previous_bar is None)
#   • the HTF bar time is ahead of the current cursor
def _waiting_for_first_closed_htf(
    cursor_time:  str | None,
    current_bar:  dict | None,
    previous_bar: dict | None,
) -> bool:
    current_time = _bar_time(current_bar)  # bar.get("time")
    return bool(
        cursor_time                        # 15m bar time is known
        and current_time                   # 4H bar exists
        and previous_bar is None           # only one closed 4H bar
        and current_time > cursor_time     # 4H bar time > 15m cursor
    )
# _none() — the none hypothesis emitter
def _none(self, reason, required_evidence, invalidation, ts, debug,
          none_sub_status=None) -> Hypothesis:
    debug = {**debug, "none_sub_status": none_sub_status} if none_sub_status else debug
    return Hypothesis(
        hypothesis_id    = self.state.hypothesis_id,
        status           = "watching",
        phase            = "none",
        direction        = "none",
        swing_alignment  = "none",
        internal_alignment = "none",
        poi_id           = None,
        poi_type         = None,
        reason           = reason,
        required_evidence = required_evidence,
        invalidation     = invalidation,
        target_policy    = "none",
        entry_policy     = "wait",
        created_at       = (
            self.state.current_hypothesis.created_at
            if self.state.current_hypothesis else ts
        ),
        updated_at       = ts,
        none_sub_status  = none_sub_status,
        debug_facts      = debug,
    )

# Note: _none() does NOT call _commit() directly.
# The caller always wraps with return self._commit(hyp).
# _commit() will NOT advance phase_episode_id for none
# (phase "none" is not in {"E","D","C","B","A"}).

08 Open Issues · Edge Cases · Next Milestone

Open Issue

Cold-start self-location only supports none → E. During warmup, Layer 3 can observe enough history to reconstruct a current HTF state that already belongs to D, C, B, or A. Layer 4 still starts with previous_phase = "none" and active_phase_e_direction = "none". The first post-warmup bar should be able to self-locate into the best current DAG node: none(warmup) → E/D/C/B/A. Current code only reliably handles none(warmup) → E.

Edge Case

warmup_waiting_for_first_closed_htf fires in two branches. The same none_sub_status is set in both the not htf branch (no structure at all) and the htf_neutral branch (structure exists but bias is neutral). A consumer cannot tell which condition caused the warmup label without reading debug_facts["htf_bias"].

Edge Case

One warmup_bars value covers very different durations. warmup_bars: 200 feeds 200 LTF bars to the lower engines and 200 HTF bars to the higher engines. The counts are independent but use the same configured integer, so they cover very different calendar spans. Reducing the value must preserve at least 14 HTF bars for liquidity ATR, but the larger risk is losing the structural context preceding a week-long sideways regime.

Edge Case

htf_resolve_unclassified is the wrong sub_status for missing E. When the classifier is in a valid HTF state (bullish open) but no D/C/B gate fires and _previous_or_active_e() is False (first bar), the fallback emits htf_resolve_unclassified. This is misleading — the real reason is "Phase E not yet committed." A dedicated sub_status like phase_e_not_yet_started would be more precise.

Edge Case

HypothesisClassifier state is NOT reset between warmup and replay. The warmup does not call classify(), so state remains at init defaults. But if the replayer is reset or re-initialised mid-session, the classifier state may carry over stale phase/shadow state into the new warmup. Always call HypothesisClassifier.__init__() (or reset state) when re-initialising the replay loop.

Fallback

Sustained htf_neutral after warmup. If none / htf_neutral persists for many bars after replay start, the structure engine did not find a CHoCH in the warmup window. Fix: increase warmup_bars, move the replay start date to a post-break bar, or verify that the pivot and structure engines received enough useful directional history. Liquidity ATR readiness is separate and can be checked independently.

Next Milestone

Add none_sub_status = "phase_e_not_yet_started" for bar 0. Currently bar 0 with valid bias but no prior Phase E emits htf_resolve_unclassified, which looks like a classifier gap. A dedicated sub_status distinguishes "first bar of a new replay" from "genuinely unclassified bar in a running session."

Next Milestone

Expose warmup health in payload. payload includes warmup_bars and engine start times in the replay metadata block, but there is no flag like "hypothesis_classifier_warmed": true/false. A downstream consumer cannot easily tell from the payload alone whether the classifier is past its silent startup window.

Edge Case

Market stays sideways across the entire warmup window. If the instrument ranges without a clean CHoCH for all warmup_bars, the structure engine reaches replay bar 0 with bias = "neutral". The selected 200-bar guardrail is intended to look through roughly a week of sideways price and preserve useful structure from before the range. If no usable break exists anywhere in the complete warmup, the instrument itself is in a genuine accumulation/distribution range and carries no directional bias worth trading. The system is correct to emit none indefinitely. none = no trade. Change the instrument, not the config.

Out of Scope

Warm the hypothesis classifier from historical data. The classifier intentionally starts fresh because its state machine (phase transitions, shadow state) is not safe to replay from arbitrary mid-stream history. Pre-warming it would require replaying the full phase history from the warmup start, which would change the warmup semantics significantly.
