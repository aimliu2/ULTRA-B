Layer 4 · Hypothesis Classifier · Phase C

Phase C — Counter Pullback

TL;DR - Phase C is the counter-pullback phase after Phase E expansion has been interrupted by D, by a direct tight-range hard pullback, by E.pullback_developing, or by failed B initiation. As coded now, C mostly reads one EC package, ltf_counter_story, then Layer 4 assigns the origin and sub-status.
→ Back to 401 Hypothesis Dataflow → Open 401 Hypothesis DAG → Related: Phase E Detail → Related: Phase D Detail

00 Core Concept

Phase C names the counter-trend pullback after the Phase E expansion leg stops being clean continuation. It is not the first reaction itself. It is the counter story that follows the reaction, with enough LTF structure to either watch for a counter POI or arm against one.

The classifier keeps the Phase E direction as prior_phase_e_direction, then Phase C trades the opposite direction. If Phase E was bullish, C is short. If Phase E was bearish, C is long.

Discretionary read chart logic

C is the moment where the pullback has a structure story of its own. The chart is no longer only saying "E stopped expanding"; it is saying "the counter leg has enough internal evidence to be tracked." 

Phase E bullish:
  E expansion up
  reaction / stall / pullback begins
  LTF counter story turns bearish
  Phase C direction = short

Phase E bearish:
  E expansion down
  reaction / stall / pullback begins
  LTF counter story turns bullish
  Phase C direction = long
Phase C is counter to the prior Phase E direction. It can be watching with a story but no selected POI, or armed when a selected counter POI exists.

01 EC Inputs

The current Phase C setup is deliberately thin. Layer 4 calls _ec_candidate(snapshot, "ltf_counter_story") and reads the resulting facts. The EC decides whether the counter story is ready; the DAG decides which C node name that story belongs to.

EC package ltf_counter_story 
Gate:
  c_gate.status == "ready"  → phase_c_story_ready

Facts consumed by Layer 4:
  selected_poi
  ltf_counter_sd_zone_ids
  ltf_counter_sd_returned_zone_ids
  ltf_bias_counter_htf
  ltf_counter_pd_break
  ltf_counter_pullback_confirmed
Status and entry policy Layer 4 output 
story_ready + no selected_poi:
  phase = C
  status = watching
  entry_policy = wait

story_ready + selected_poi:
  phase = C
  status = armed
  entry_policy = hybrid only if pullback trading is enabled

otherwise:
  no Phase C promotion from this gate
EC should report counter evidence. It should not decide whether the node is C.hard_pullback, C.slow_pullback, or C.htf_reaction_pullback. That naming belongs to Layer 4 because it depends on DAG origin.

02 Current C Nodes

The DAG currently names five Phase C variants. Some are implemented directly in the classifier. Others are present in the DAG note as intended vocabulary and need cleanup as migration continues.

C.htf_reaction_pullback D → C

This is the regular C node after a Phase D reaction. The classifier uses _phase_c_sub_status_from_current_d() and currently maps reaction-point, HTF zone reclaim, HTF P/D reclaim, and HTF EQ reclaim variants into htf_reaction_pullback. 

Entry family:
  D.reaction_point
  D.htf_pd_grab_reclaim_test
  D.htf_eq_grab_reclaim_test

Required:
  phase_c_story_ready
  phase_d_reaction_confirmed
C.hard_pullback E fast path

The implemented hard path is the tight P/D shortcut from Phase E. It fires when Phase E context is ready, the HTF P/D range stopped expanding, the LTF counter orderflow is clean and unbroken, and the pullback depth reaches the midpoint. 

phase_c_origin_node = E.fast_hard_pullback
phase_c_sub_status = hard_pullback
phase_c_selection_reason = fast_midpoint_pullback_after_phase_e_expansion
phase_c_quality = choppy_tight_range_fast_pullback
threshold = ltf_pullback_depth_pct >= 50
C.slow_pullback E.pullback_developing → C

The slow path currently fires when the Phase E shadow is already E.pullback_developing and LTF orderflow has confirmed a full counter-direction BoS/MSS — Case F level: confirmed_direction == counter-HTF and regime == "directional". Because E.pullback_developing has been corrected back to an orderflow-MSS trigger, this path needs another review to avoid using the same orderflow verdict twice as both shadow entry and C maturity. 

Gate:
  previous_phase != "B"
  phase_e_shadow_node == "E.pullback_developing"
  ltf_counter_bos_confirmed == True            ← Case F: confirmed_direction + regime directional

EC fact (ltf_counter_story.debug_facts):
  ltf_counter_bos_confirmed =
    orderflow.confirmed_direction == counter-HTF
    and orderflow.regime == "directional"

phase_c_origin_node = E.pullback_developing
phase_c_sub_status = slow_pullback
phase_c_selection_reason =
  ltf_pd_flipped_counter_after_failed_e_continuation_attempt

The failed pro-attempt is implicit: if we are still in E.pullback_developing when the BoS fires, Phase D never committed. If D had committed, the DAG would have left Phase E and the shadow would have reset.

C.after_inducement / C.no_followthrough B failure → C

When B.initiation_watch fails, the classifier can resolve back into Phase C. The current code emits pullback.no_followthrough when the B source anchor is run or when the B decision fails with LTF counter confirmation. It emits pullback.after_inducement when B initiation is rejected by HTF reaction plus LTF counter flow. 

B.initiation_watch → C.pullback.no_followthrough
B.initiation_watch → C.pullback.after_inducement

03 DAG Transitions

Current DAG intent says Phase C can arrive from D, from fast Phase E pullback, from mature Phase E pullback, or from failed B initiation. Once C exists, the usual next step is B reclaim permission, while failed reaction can collapse back to E.

Entry families DAG 
D.reaction_point              → C.htf_reaction_pullback
D.htf_pd_grab_reclaim_test    → C.htf_reaction_pullback
D.htf_eq_grab_reclaim_test    → C.htf_reaction_pullback

E.seeking                     → C.hard_pullback
E.stalling                    → C.hard_pullback
E.pullback_developing         → C.hard_pullback
E.pullback_developing         → C.slow_pullback

B.initiation_watch failure    → C.pullback.no_followthrough
B.initiation_watch rejection  → C.pullback.after_inducement
Exit families C → B / D / E 
C → B.initiation_watch
C → B.strict_reclaim
C → B.shallow_reclaim

C → D.* if a new reaction/liquidity boundary appears from current C
C → E.seeking if the prior Phase E direction resumes and reaction fails
The DAG must preserve origin. A hard pullback from a tight Phase E range is not the same historical object as a C node produced after D, even if both use the same ltf_counter_story facts.

04 Algorithm

These snippets show the current classifier shape. Phase C setup is generic; the branch that calls it determines the C sub-status and origin debug fields. 

# classify(), simplified
if previous_phase == "D":
    phase_c = self._phase_c_setup(snapshot, ltf, direction)
    if phase_c["phase_c_story_ready"] and self._phase_d_reaction_confirmed(phase_c):
        hyp = self._phase_c(..., phase_sub_status="htf_reaction_pullback")
        return self._commit(hyp)

phase_d = self._phase_d_setup(...)
if phase_d["phase_d_ready"]:
    return self._commit(self._phase_d(...))

phase_c_from_e = self._phase_c_setup(snapshot, ltf, direction)
fast_hard = self._phase_c_fast_hard_pullback_setup(debug, phase_c_from_e)
if previous_phase != "B" and fast_hard["phase_c_fast_hard_pullback_ready"]:
    hyp = self._phase_c(..., phase_sub_status="hard_pullback")
    return self._commit(hyp)

if previous_phase != "B" and phase_e_shadow_node == "E.pullback_developing":
    if phase_c_from_e["phase_c_ltf_counter_bos_confirmed"]:  # Case F: confirmed_direction + regime directional
        hyp = self._phase_c(..., phase_sub_status="slow_pullback")
        return self._commit(hyp)
def _phase_c_setup(self, snapshot, ltf, prior_direction):
    c_gate, facts = _ec_candidate(snapshot, "ltf_counter_story")
    selected_poi = facts.get("selected_poi")
    story_ready = c_gate.get("status") == "ready"
    armed = bool(story_ready and selected_poi)

    return {
        "phase_c_story_ready": story_ready,
        "phase_c_status": "armed" if armed else ("watching" if story_ready else None),
        "phase_c_direction": "short" if prior_direction == "long" else "long",
        "phase_c_counter_zone_direction": counter_zone_direction,
        "phase_c_selected_poi": selected_poi,
        "phase_c_ltf_bias_counter_htf": bool(facts.get("ltf_bias_counter_htf")),
        "phase_c_ltf_counter_pd_break": bool(facts.get("ltf_counter_pd_break")),
        "phase_c_ltf_counter_pullback_confirmed": bool(facts.get("ltf_counter_pullback_confirmed")),
    }
def _phase_c(self, prior_direction, selected_poi, status, ts, debug, phase_sub_status=None):
    trade_direction = "short" if prior_direction == "long" else "long"
    entry_policy = (
        "hybrid"
        if status == "armed"
        and (self.config.get("allow_pullback_trade")
             or self.config.get("phase_c_allow_pullback_trade"))
        else "skip"
    )
    if status == "watching":
        entry_policy = "wait"

    return Hypothesis(
        phase="C",
        status=status,
        direction=trade_direction,
        swing_alignment="counter_swing",
        internal_alignment="pro_internal",
        phase_sub_status=phase_sub_status,
        entry_policy=entry_policy,
    )
ready = bool(
    debug.get("phase_e_context_status") == "ready"
    and not debug.get("phase_e_context_new_htf_extreme")
    and debug.get("phase_e_context_htf_pd_stopped_expanding")
    and debug.get("phase_e_context_ltf_counter_orderflow_clean")
    and not debug.get("phase_e_context_ltf_counter_orderflow_broken")
    and depth_value is not None
    and depth_value >= 50.0
)

06 Open Issues

This page only lists Phase C issues. Completed Phase E migration items stay on the Phase E note.

Archived · Completed

Slow C maturity after Phase E MSS watch. Resolved boundary: E.pullback_developing enters on ltf_counter_orderflow_mss_watch when the protected LTF orderflow anchor breaks and the MSS monitor opens. C.slow_pullback remains the later maturity step: E.pullback_developing + ltf_counter_bos_confirmed (Case F: confirmed_direction + regime == directional).

Phase C Boundary

Choose the stream by job. Structural ChoCh is the fresh one-bar chronology signal for D/liquidity evidence. Orderflow BoS / confirmed_direction is the later swing-sequence maturity signal for C.slow_pullback. Do not substitute structural ChoCh for the slow-C maturity gate.

Open Issue

Phase C quality language is incomplete. phase_c_quality is set on the fast hard path (choppy_tight_range_fast_pullback), but the D-to-C and slow E-to-C paths carry no equivalent quality field. Debug comparison between C variants is uneven until all paths emit a quality tag.

Edge Case

Cold-start C.hard_pullback does not set active_phase_e_direction. When the fast path fires without a prior E classification, _phase_e() is never called and active_phase_e_direction stays "none". The C carry block guards on active_phase_e_direction == direction; it fails on the next bar and falls through to a fresh E classification. The test at line 717 is single-snapshot and does not cover this. Known: cold-start beyond none → E is in-flight in CLAUDE.md.

Edge Case

Watching C can exist without a selected POI. This is probably correct, but it means a strong counter story is visible in Layer 4 while Layer 5 has no tradeable object. Replayer UI should make the difference between watching and armed visually clear.

Archived · Resolved

E.pullback_developing → E.stalling disruption exit is retired. The current decision is intentional: after the first counter-orderflow MSS moves Phase E shadow into E.pullback_developing, broken/pro LTF orderflow does not send it back to E.stalling. The shadow holds until a new HTF extreme resets Phase E, or Layer 4 selects D/C. Legacy disruption debug keys have been removed from the runtime output.

Next Milestone

Reconcile Phase C with the Phase E shadow migration. C should consume cleanly named origin memory from Phase E/D/B rather than inferring from a single generic ltf_counter_story gate. Run the full suite to confirm Q3: no stale EC fact-shape assertions in D→C and C-hold tests.
