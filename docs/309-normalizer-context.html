Layer 3 · Context · Snapshot Normalizer

Snapshot Normalizer - ContextSnapshot Projection

TL;DR - Snapshot Normalizer is the read-only Layer 3 projection step. It reads channel-owned semantic state after Structure, S/D, Liquidity, and Orderflow update their stores, then prints the current ContextSnapshot for this bar. It must stay boring: facts, relations, nearest objects, and lifecycle flags only.

Back to 301 Layer 3 Context

00 Core Concept

Snapshot Normalizer is not another detector and not a thesis engine. It is the adapter between channel-owned state and the compact surface consumed by Fusion, Evidence Compiler, the replayer UI, and later Layer 4.

State Store / channel engines
-> Snapshot Normalizer
-> ContextSnapshot
-> Fusion
-> Evidence Compiler
-> Layer 4
Step 1 · Read channel state after detectors

Structure, S/D Zone, Liquidity, and Orderflow own their lifecycles. Snapshot Normalizer reads the current state after those writers finish for the bar.

Step 2 · Add cursor relations current bar

The runtime cursor supplies current timestamp, price, bar index, and timeframe. Snapshot Normalizer uses that cursor to calculate distance, P/D position, probe relation, and invalidation proximity.

Step 3 · Project literal facts no thesis

The output says what is true now. It does not decide Phase B, Phase D, entry readiness, evidence strength, or hypothesis switching.

If a field sounds like a thesis node, it probably belongs to Evidence Compiler or Layer 4 instead of Snapshot Normalizer.

01 Inputs

Snapshot Normalizer reads factual context channels. Each channel owns its own state and exposes enough snapshot material for projection.

Structure Context

Structure provides bias, bias tier, P/D range state, pullback confirmation, structural anchors, and the live anchor probe. These are structural facts, not Orderflow conclusions.

structure:
  bias
  biasTier
  rangeState
  currentLeg
  range_high / range_low
  pd_value_pct
  range_position_pct
  structure_sequence
  structure_probe

S/D Context

S/D Context provides active supply and demand zones, zone state, geometry, owner timeframe, and last resolved zone. Snapshot Normalizer sorts the relevant zones by current-price distance.

nearestSupplyZones[]
nearestDemandZones[]
lower_last_resolved_zone
higher_last_resolved_zone

demand distance = current_price - zone.high
supply distance = zone.low - current_price

Liquidity Context

Liquidity Context provides HTF P/D, EQ, and ITR pool lifecycle facts. Snapshot Normalizer language stays factual: grab/reclaim readiness, source, side, direction, level, and trajectory.

active_htf_pd_pools[]
active_htf_eq_pools[]
active_htf_itr_pools[]
event_log[]

htf_pd_grab_reclaim_ready
htf_eq_grab_reclaim_ready
htf_itr_grab_reclaim_ready
htf_itr_anchor_run_ready
Snapshot Normalizer says HTF P/D boundary grab/reclaim facts. Evidence Compiler may later call that a candidate pattern. Layer 4 may later map it to a B or D hypothesis node.

Orderflow Context

Orderflow is the migration target for useful legacy Structural Attempt language. Structure owns anchor facts; Orderflow owns current pressure, regime, MSS monitor state, protected anchor, decision point, and probe relation. Orderflow choch_* aliases are retired; consumers should read mss_* fields directly.

02 Snapshot

The ContextSnapshot is the current tape print for one bar. It should be stable enough for the replayer/UI and compact enough that Layer 4 does not need to rebuild raw evidence sequences.

ContextSnapshot:
  currentTimestamp
  currentPrice
  currentBarIndex
  timeframe

  bias
  biasTier
  rangeState
  currentLeg
  pdPosition

  lastConfirmedHigh
  lastConfirmedLow
  keyLevelsAbove[]
  keyLevelsBelow[]
  nearestSupplyZones[]
  nearestDemandZones[]

  structure
  liquidity
  orderflow
  invalidationRefs[]

The current price is not a Layer 2 event. It is the runtime cursor value used by Layer 3 projection. Layer 2 emits primitive CE / PE events; Layer 3 combines those event records with the current bar heartbeat to update and project semantic context.

03 Ownership

The boundary is the main point of this note. Snapshot Normalizer owns projection; it does not own interpretation.

Snapshot Normalizer answers:
  What is currently true on this bar?
  Which objects are active?
  Which lifecycle flags are ready?
  Which levels / zones / pools / anchors are nearest or relevant?
  What is the current price relation to those objects?

Snapshot Normalizer does not answer:
  Is this Phase B?
  Is this Phase D?
  Is this an entry?
  Is this evidence strong enough?
  Should Layer 4 switch hypothesis node?
Example handoff literal to qualified 
Liquidity engine:
  pool.status = liquidity_grab_confirmed

Snapshot Normalizer:
  htf_pd_grab_reclaim_ready = true
  htf_pd_grab_reclaim_level = 1.08500
  htf_pd_grab_reclaim_direction = bearish

Evidence Compiler:
  candidate.pattern = grab_reclaim_at_htf_pd_boundary
  candidate.location_context = htf_pd_boundary
  candidate.relation_to_probe = reclaiming

Layer 4:
  may map candidate to D.htf_pd_grab_reclaim_test.pending_outcome

04 Algorithm

The implementation target is to extract the projection work currently embedded in runtime snapshots into a dedicated read-only module. Existing payload keys can remain for UI compatibility while context_snapshot becomes the normalized surface.

# proposed: SnapshotNormalizer.project()
def project(cursor, channels) -> ContextSnapshot:
    structure = channels.structure.snapshot(cursor.price)
    zones = channels.sd.snapshot(cursor.price)
    liquidity = channels.liquidity.snapshot()
    orderflow = channels.orderflow.snapshot()

    return ContextSnapshot(
        currentTimestamp=cursor.timestamp,
        currentPrice=cursor.close,
        currentBarIndex=cursor.index,
        timeframe=cursor.timeframe,
        structure=structure,
        liquidity=liquidity,
        orderflow=orderflow,
        nearestSupplyZones=nearest_supply(zones, cursor.close),
        nearestDemandZones=nearest_demand(zones, cursor.close),
        invalidationRefs=nearest_invalidation_refs(structure, orderflow),
    )
# zone distance projection
def project_zone_distance(zone, current_price):
    if zone.direction == "demand":
        return current_price - zone.high

    if zone.direction == "supply":
        return zone.low - current_price

    return None
# runtime compatibility during migration
payload = legacy_snapshot_payload(...)
payload["context_snapshot"] = normalizer.project(cursor, channels)

# Existing UI payload keys stay alive until the replayer reads context_snapshot.
payload["structure"] = payload["context_snapshot"]["structure"]
payload["liquidity"] = payload["context_snapshot"]["liquidity"]
payload["orderflow"] = payload["context_snapshot"]["orderflow"]

05 Config

There is no committed standalone snapshot_normalizer: config block yet. The current relevant config is the channel source policy Snapshot Normalizer projects, especially Structure sequence retention for Orderflow.

# ultrab/src/ultrab/replayer/config.yaml
structure:
  enabled: true
  tier: "itr"
  structure_sequence_limit: 8
  dual_display: dual

# proposed future scope
orderflow:
  probe_points: 8
  source_policy:
    lower_timeframe: structure_sequence
    higher_timeframe: itr_sequence
# current runtime snapshot shape is still inline
def snapshot(self, classify=True):
    lower_price = self._current_lower_price()
    higher_price = self._current_higher_price()
    lower_zones = self.lower_sd_zone.get_zone_snapshot(lower_price)
    higher_zones = self.higher_sd_zone.get_zone_snapshot(higher_price)
    lower_structure = self.lower_structure.get_snapshot(lower_price)
    higher_structure = self.higher_structure.get_snapshot(higher_price)
    liquidity = self.liquidity.snapshot()

    payload = {
        "zones": lower_zones + higher_zones,
        "lower_structure": lower_structure,
        "higher_structure": higher_structure,
        "liquidity": liquidity,
    }

06 Resolved / Open Issues

These are the out-of-order or conflicting points found while consolidating the note with the current backend and the Orderflow discussion. Completed items stay here as archived decisions so the migration path remains visible.

Archived · Completed

Orderflow enters through 09. The replayer now emits context_snapshot, and backend Orderflow is plugged through SnapshotNormalizer instead of being bolted directly into the old snapshot body.

Archived · Completed

Projection extracted from snapshots. ReplaySession.snapshot() and DualSMCRuntime.snapshot() still serve adapter payloads, but the shared projection responsibility now lives in SnapshotNormalizer.project(). Existing UI keys remain for compatibility.

Fallback

Migrate consumers gradually. The compatibility payloads are intentionally still present: structure, zones, liquidity, orderflow, and dual-mode keys. Replayer UI, Evidence Compiler, and future Layer 4 consumers should move toward context_snapshot over time.

Out of Scope

Do not score evidence here. Sideway Orderflow can expose weak pressure, compression, or MSS watch, but confidence weighting and candidate qualification belong to Evidence Compiler. Layer 4 still owns thesis meaning.
